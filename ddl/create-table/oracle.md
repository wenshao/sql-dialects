## 基本建表

一个典型的业务表，涵盖 Oracle 建表的核心特性
```sql
CREATE TABLE users (
    id         NUMBER(19)    GENERATED ALWAYS AS IDENTITY PRIMARY KEY,  -- 12c+
    username   VARCHAR2(64 CHAR)  NOT NULL,      -- CHAR 语义，见下文
    email      VARCHAR2(255 CHAR) NOT NULL,
    age        NUMBER(3),
    balance    NUMBER(10,2)  DEFAULT 0.00,
    bio        CLOB,
    created_at TIMESTAMP     DEFAULT CURRENT_TIMESTAMP NOT NULL,
    updated_at TIMESTAMP     DEFAULT CURRENT_TIMESTAMP NOT NULL,
    CONSTRAINT uk_users_username UNIQUE (username),
    CONSTRAINT uk_users_email UNIQUE (email)
);
```

Oracle 没有 ON UPDATE CURRENT_TIMESTAMP，需要触发器
```sql
CREATE OR REPLACE TRIGGER trg_users_updated_at
    BEFORE UPDATE ON users
    FOR EACH ROW
BEGIN
    :NEW.updated_at := CURRENT_TIMESTAMP;
END;
/
```

设计要点:
  1. GENERATED ALWAYS AS IDENTITY: 12c+ 推荐的自增方式（底层仍是 SEQUENCE）
  2. VARCHAR2(64 CHAR): 显式指定 CHAR 语义，避免多字节截断（见下文详解）
  3. NUMBER(19): 等价于 Java 的 long 范围，适合做主键
  4. CLOB: 大文本，最大 4GB。注意 CLOB 不能直接用 = 比较，需要 DBMS_LOB.COMPARE
  5. Oracle 中每条 DML 自动开启事务，不需要 BEGIN TRANSACTION

## VARCHAR2(BYTE) vs VARCHAR2(CHAR): 沉默的截断陷阱

这是 Oracle 中最容易让新手踩坑的类型问题

VARCHAR2(64)  -- 不带限定符时，取决于 NLS_LENGTH_SEMANTICS 参数
VARCHAR2(64 BYTE)  -- 明确：最多存 64 字节
VARCHAR2(64 CHAR)  -- 明确：最多存 64 个字符

为什么这很重要？
  在 AL32UTF8 编码下，一个中文字符占 3 字节，一个 emoji 占 4 字节
  VARCHAR2(64 BYTE) 只能存 21 个中文字符！
  如果你认为你定义了 64 个字符的列，实际上可能只有 21 个字符就截断了

NLS_LENGTH_SEMANTICS 默认是 BYTE，很多 DBA 不会改这个参数
最佳实践: 始终显式写 VARCHAR2(n CHAR)，不要依赖数据库默认设置

> **注意**: VARCHAR2 最大长度：
  标准 SQL: 4000 字节
  12c+ MAX_STRING_SIZE=EXTENDED: 32767 字节
  修改 MAX_STRING_SIZE 需要停机且不可逆，慎重考虑

## NUMBER vs BINARY_DOUBLE: 精度 vs 速度

NUMBER(p,s): Oracle 的通用数值类型，十进制精度，最大 38 位
  存储: 变长，1-22 字节
  精度: 精确，适合金融计算
  性能: 比 IEEE 浮点慢（软件运算而非硬件 FPU）

BINARY_DOUBLE: IEEE 754 双精度浮点
  存储: 固定 8 字节
  精度: 约 15-16 位有效数字，有舍入误差
  性能: 硬件 FPU 运算，科学计算比 NUMBER 快 3-10 倍

BINARY_FLOAT: IEEE 754 单精度浮点，4 字节，约 7 位有效数字

选择指南:
  金融/货币 → NUMBER(p,s)，永远不要用浮点
  科学计算/机器学习 → BINARY_DOUBLE
  传感器数据/日志 → BINARY_FLOAT（节省空间，精度够用）

## '' = NULL: Oracle 最大的坑

在 Oracle 中，空字符串 '' 等同于 NULL。这违反 SQL 标准，也是无数 bug 的根源

示例：这些条件永远返回 FALSE
  WHERE name = ''     →  等价于 WHERE name = NULL  →  结果: 无匹配
  WHERE name != ''    →  等价于 WHERE name != NULL →  结果: 无匹配
  WHERE '' = ''       →  等价于 WHERE NULL = NULL  →  结果: FALSE

实际 bug 场景:
  INSERT INTO t (name) VALUES ('');   -- 实际插入 NULL
  SELECT COUNT(*) FROM t WHERE name IS NOT NULL;  -- 不包含上面那行
  SELECT * FROM t WHERE name = '';   -- 永远返回 0 行！

从其他数据库迁移到 Oracle 时，所有空字符串比较逻辑都需要重写
最佳实践: 在 Oracle 中统一用 IS NULL / IS NOT NULL，避免 = '' 和 != ''

## IDENTITY (12c+) vs SEQUENCE: 迁移路径

方式一: IDENTITY 列 (12c+, 推荐新项目使用)
```sql
CREATE TABLE orders_v1 (
    id NUMBER(19) GENERATED ALWAYS AS IDENTITY
       (START WITH 1 INCREMENT BY 1 CACHE 1000 NOORDER),
    order_no VARCHAR2(32 CHAR) NOT NULL,
    CONSTRAINT pk_orders_v1 PRIMARY KEY (id)
);
```

GENERATED ALWAYS: 不允许手动指定 ID（INSERT 时不能指定 id 列）
GENERATED BY DEFAULT: 允许手动指定，不指定时自动生成
GENERATED BY DEFAULT ON NULL: 手动指定 NULL 时也自动生成（最灵活）
CACHE 1000: 预分配 1000 个序列值到内存，RAC 环境下大幅减少争用

方式二: 传统 SEQUENCE（所有版本通用）
```sql
CREATE SEQUENCE seq_orders START WITH 1 INCREMENT BY 1 CACHE 1000;
CREATE TABLE orders_v2 (
    id NUMBER(19) DEFAULT seq_orders.NEXTVAL NOT NULL,  -- 12c+ 可以用 DEFAULT
    order_no VARCHAR2(32 CHAR) NOT NULL,
    CONSTRAINT pk_orders_v2 PRIMARY KEY (id)
);
```

12c 之前需要触发器来调用 NEXTVAL:
```sql
CREATE TRIGGER trg_orders_id BEFORE INSERT ON orders_v2
```

  FOR EACH ROW BEGIN :NEW.id := seq_orders.NEXTVAL; END;

IDENTITY 底层就是 SEQUENCE + 隐式列默认值
但 SEQUENCE 更灵活: 多表共享、手动获取下一个值、批量预分配

## IOT (Index-Organized Table) vs 堆表

堆表 (Heap Table): 默认，数据无序存储，通过 ROWID 访问
IOT: 数据按主键排序存储在 B+ 树中，类似 MySQL InnoDB 的聚簇索引

```sql
CREATE TABLE lookup_codes (
    category   VARCHAR2(30 CHAR) NOT NULL,
    code       VARCHAR2(30 CHAR) NOT NULL,
    label      VARCHAR2(100 CHAR) NOT NULL,
    sort_order NUMBER(3) DEFAULT 0,
    CONSTRAINT pk_lookup_codes PRIMARY KEY (category, code)
) ORGANIZATION INDEX                      -- IOT
  OVERFLOW TABLESPACE users_data;         -- 溢出段: 行太长时非键列存到这里

-- IOT 适用场景:
--   1. 查找表/字典表: 几乎总是按主键查询
--   2. 时序数据: 按时间主键范围扫描
--   3. 窄表: 列少且总是按主键访问
-- IOT 不适用:
--   1. 频繁的全表扫描
--   2. 大量非主键索引访问（二级索引需要逻辑 ROWID 猜测 + 物理 ROWID 回退）
--   3. 频繁的 DML（B+ 树维护开销大）
```

## 表空间规划

生产环境绝不要把所有东西放 USERS 表空间
```sql
CREATE TABLE large_events (
    id         NUMBER(19)    GENERATED ALWAYS AS IDENTITY,
    event_type VARCHAR2(50 CHAR) NOT NULL,
    payload    CLOB,
    created_at TIMESTAMP     DEFAULT CURRENT_TIMESTAMP NOT NULL,
    CONSTRAINT pk_large_events PRIMARY KEY (id)
        USING INDEX TABLESPACE idx_ts,    -- 索引放专用表空间
    LOB (payload) STORE AS SECUREFILE (
        TABLESPACE lob_ts                 -- LOB 数据放专用表空间
        ENABLE STORAGE IN ROW             -- 小于 4000 字节的 LOB 内联存储
        COMPRESS HIGH                     -- SecureFile LOB 压缩 (11g+)
    )
) TABLESPACE data_ts;                     -- 表数据放专用表空间

-- 表空间规划原则:
--   data_ts: 业务表数据
--   idx_ts:  索引（I/O 特性不同，可以放不同磁盘）
--   lob_ts:  LOB 数据（通常很大，需要独立管理）
--   temp_ts: 临时段（排序、哈希连接）
--   undo_ts: 回滚段
```

## 压缩与批量加载优化

COMPRESS + NOLOGGING 组合，ETL / 数据仓库场景必备

```sql
CREATE TABLE fact_sales (
    sale_id    NUMBER(19)     NOT NULL,
    product_id NUMBER(10)     NOT NULL,
    amount     NUMBER(12,2)   NOT NULL,
    sale_date  DATE           NOT NULL,
    CONSTRAINT pk_fact_sales PRIMARY KEY (sale_id)
) COMPRESS FOR OLTP                       -- 11g+: OLTP 压缩（所有 DML 操作均压缩）
  NOLOGGING;                              -- 不写 redo log，大幅加速批量加载

-- 压缩选项对比:
--   COMPRESS BASIC:       只压缩直接路径加载 (INSERT /*+ APPEND */)
--   COMPRESS FOR OLTP:    所有 DML 都压缩（11gR2+ Advanced Compression 许可）
--   COMPRESS FOR QUERY:   列式压缩，Exadata 专用（HCC - Hybrid Columnar Compression）
--   COMPRESS FOR ARCHIVE: 最高压缩比，Exadata 专用
--
-- NOLOGGING 注意事项:
--   1. 不写 redo → Data Guard 备库数据不一致
--   2. 加载完成后建议: ALTER TABLE fact_sales LOGGING;
--   3. 然后做一次全备份
--   4. FORCE LOGGING 模式的数据库会忽略 NOLOGGING
```

## 分区表

```sql
CREATE TABLE audit_logs (
    id         NUMBER(19)     GENERATED ALWAYS AS IDENTITY,
    action     VARCHAR2(50 CHAR)  NOT NULL,
    user_id    NUMBER(19),
    details    CLOB,
    created_at TIMESTAMP      DEFAULT CURRENT_TIMESTAMP NOT NULL,
    CONSTRAINT pk_audit_logs PRIMARY KEY (id, created_at)
) PARTITION BY RANGE (created_at) (
    PARTITION p_2024_q1 VALUES LESS THAN (TIMESTAMP '2024-04-01 00:00:00'),
    PARTITION p_2024_q2 VALUES LESS THAN (TIMESTAMP '2024-07-01 00:00:00'),
    PARTITION p_2024_q3 VALUES LESS THAN (TIMESTAMP '2024-10-01 00:00:00'),
    PARTITION p_2024_q4 VALUES LESS THAN (TIMESTAMP '2025-01-01 00:00:00'),
    PARTITION p_max     VALUES LESS THAN (MAXVALUE)
) ENABLE ROW MOVEMENT;                    -- 允许 UPDATE 分区键时行跨分区移动

-- 11g+: INTERVAL 分区，自动创建分区
CREATE TABLE metric_data (
    id         NUMBER(19)     GENERATED ALWAYS AS IDENTITY,
    metric_key VARCHAR2(100 CHAR) NOT NULL,
    value      NUMBER,
    ts         TIMESTAMP      NOT NULL
) PARTITION BY RANGE (ts)
  INTERVAL (NUMTOYMINTERVAL(1, 'MONTH'))  -- 每月自动创建新分区
  (PARTITION p_init VALUES LESS THAN (TIMESTAMP '2024-01-01 00:00:00'));
```

## Edition-Based Redefinition (EBR) - 零停机 DDL

11gR2+: 允许在不停机的情况下修改表结构
原理: 通过 editioning view 间接访问表，新旧版本代码可以同时运行

步骤概要:
  1. ALTER USER myapp ENABLE EDITIONS;
  2. CREATE EDITION v2 AS CHILD OF ora$base;
  3. 在新 edition 中修改 editioning view 和触发器
  4. 新会话连接到 v2 edition，旧会话保持在 ora$base
  5. 数据交叉同步触发器处理新旧格式差异

这是 Oracle 企业级在线升级的核心技术，但学习曲线陡峭
替代方案: 12c 的 DBMS_REDEFINITION 在线重定义（更常用，但仍需停机窗口做交换）

## 虚拟列与函数索引

```sql
CREATE TABLE products (
    id         NUMBER(19) GENERATED ALWAYS AS IDENTITY PRIMARY KEY,
    name       VARCHAR2(200 CHAR) NOT NULL,
    price      NUMBER(10,2) NOT NULL,
    tax_rate   NUMBER(5,4) DEFAULT 0.13,
    -- 11g+: 虚拟列，不占存储空间，查询时计算
    total_price NUMBER(12,2) GENERATED ALWAYS AS (price * (1 + tax_rate)) VIRTUAL,
    -- 可以对虚拟列建索引
    CHECK (price >= 0)
);
CREATE INDEX idx_products_total ON products(total_price);
```

## 全局临时表 vs 私有临时表

全局临时表（所有版本）: 结构永久，数据会话/事务级别
```sql
CREATE GLOBAL TEMPORARY TABLE gtt_session_data (
    id   NUMBER,
    data VARCHAR2(200 CHAR)
) ON COMMIT PRESERVE ROWS;    -- 数据保留到会话结束
-- ON COMMIT DELETE ROWS: 数据在 COMMIT 时自动删除

-- 18c+: 私有临时表（不需要预定义结构）
-- CREATE PRIVATE TEMPORARY TABLE ora$ptt_temp AS SELECT * FROM orders;
-- 会话结束自动销毁，表名必须以 ORA$PTT_ 开头（可配置前缀）
```

## 版本演进总结

8i (1999):  分区表、IOT、Java 存储过程
9i (2001):  MERGE 语句、外部表、XML DB
10g (2003): 闪回技术全家桶、ASM、COMPRESS BASIC、回收站
11g (2007): SecureFile LOB、虚拟列、INTERVAL 分区、结果缓存、EBR
12c (2013): IDENTITY 列、VARCHAR2 32K、多租户 (CDB/PDB)、Adaptive Plans
18c (2018): 私有临时表、多态表函数（版本号跳到年份）
19c (2019): 自动索引、实时统计信息、长期支持版（LTS）
21c (2021): JSON 原生数据类型、区块链表、不可变表、SQL 宏
23c (2023): JSON 关系二元性、布尔类型、SCHEMA 级权限、IF [NOT] EXISTS 语法
            GROUP BY 列别名、FROM 子句在 SELECT 之前、UPDATE/DELETE RETURNING 增强
