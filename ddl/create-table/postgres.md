## 基本建表（现代写法，PostgreSQL 10+）

一个典型的业务表，包含了 PostgreSQL 建表中推荐的最佳实践
```sql
CREATE TABLE users (
    id         BIGINT       GENERATED ALWAYS AS IDENTITY PRIMARY KEY,
    username   TEXT         NOT NULL UNIQUE,
    email      TEXT         NOT NULL UNIQUE,
    age        INTEGER      CHECK (age > 0 AND age < 200),
    balance    NUMERIC(10,2) DEFAULT 0.00,
    bio        TEXT,
    created_at TIMESTAMPTZ  NOT NULL DEFAULT now(),
    updated_at TIMESTAMPTZ  NOT NULL DEFAULT now()
);
```

设计要点（每一条都有原因，不是随便选的）:

  1. GENERATED ALWAYS AS IDENTITY 而非 SERIAL:
     SERIAL 本质是创建一个 sequence + 设 DEFAULT，有几个坑:
       - 权限问题: GRANT INSERT 不会自动授权底层 sequence，需要单独 GRANT USAGE
       - pg_dump 问题: 不能正确跟踪 sequence 的 ownership
       - 可被绕过: 用户可以手动指定 id 值，导致后续自增冲突
     IDENTITY (SQL 标准语法，10+ 支持) 解决了所有这些问题:
       - GENERATED ALWAYS: 禁止手动指定值（除非 OVERRIDING SYSTEM VALUE）
       - GENERATED BY DEFAULT: 允许手动指定（行为类似 SERIAL，但更规范）

  2. TEXT 而非 VARCHAR(n):
     PostgreSQL 官方文档明确说: "在这三种类型之间没有性能差异"（指 CHAR, VARCHAR, TEXT）
     VARCHAR(n) 唯一的作用是增加一个长度检查约束，如果你不需要这个约束，TEXT 更简洁
     TEXT 的存储和性能与 VARCHAR 完全一样，不像 MySQL 那样 TEXT 有特殊存储机制
     如果确实需要长度限制，用 CHECK (length(col) <= 100) 比 VARCHAR(100) 更灵活
     （因为 ALTER TABLE 改 VARCHAR 长度在早期版本需要重写表，CHECK 约束不需要）

  3. TIMESTAMPTZ 而非 TIMESTAMP:
     这是 PostgreSQL 最容易踩的坑之一，记住一条规则: 永远用 TIMESTAMPTZ
     TIMESTAMP (without time zone): 存什么就是什么，不做任何时区处理
     TIMESTAMPTZ (with time zone): 存储为 UTC，显示时按 session timezone 转换
     两者占用空间完全一样（都是 8 字节），没有任何性能差异
     用 TIMESTAMP 的唯一合理场景: 存储与时区无关的抽象时间（如"每天 09:00 开会"）

  4. now() 而非 CURRENT_TIMESTAMP:
     功能完全一样，now() 更短。两者都返回事务开始时间（不是语句执行时间）
     如果需要语句执行时间，用 clock_timestamp()

  5. CHECK 约束: PostgreSQL 从一开始就正确支持 CHECK（不像 MySQL 到 8.0.16 才支持）

PostgreSQL 没有 ON UPDATE CURRENT_TIMESTAMP，需要触发器
```sql
CREATE OR REPLACE FUNCTION update_updated_at()
RETURNS TRIGGER AS $$
BEGIN
    NEW.updated_at = now();
    RETURN NEW;
END;
$$ LANGUAGE plpgsql;

CREATE TRIGGER trg_users_updated_at
    BEFORE UPDATE ON users
    FOR EACH ROW
    EXECUTE FUNCTION update_updated_at();
```

> **注意**: EXECUTE FUNCTION 是 11+ 语法，之前用 EXECUTE PROCEDURE（仍可用但已废弃）

## SERIAL vs IDENTITY 对比（重要！）

旧写法（不推荐，但大量遗留代码仍在使用）
```sql
CREATE TABLE legacy_table (
    id SERIAL PRIMARY KEY   -- 其实是: CREATE SEQUENCE + DEFAULT nextval(...)
);
```

SERIAL 的问题:
  1. 不是真正的类型，是宏展开: SERIAL -> INTEGER + SEQUENCE + DEFAULT
  2. BIGSERIAL -> BIGINT + SEQUENCE，SMALLSERIAL -> SMALLINT + SEQUENCE
  3. DROP TABLE 不一定删除关联的 sequence（取决于 ownership 是否正确设置）
  4. INSERT INTO t (id) VALUES (999) 不会报错，但 sequence 不知道，下次自增可能冲突

新写法（推荐，10+）
```sql
CREATE TABLE modern_table (
    id BIGINT GENERATED ALWAYS AS IDENTITY PRIMARY KEY
    -- GENERATED ALWAYS: INSERT 时不能指定 id 值（会报错）
    -- 如果真要覆盖: INSERT INTO t (id, ...) OVERRIDING SYSTEM VALUE VALUES (999, ...)
);
```

迁移建议: 新项目一律用 IDENTITY，老项目不需要特意迁移，但新表用 IDENTITY

## UUID 作为主键

13+ 内置 gen_random_uuid()，不再需要 pgcrypto 或 uuid-ossp 扩展
```sql
CREATE TABLE orders (
    id         UUID         DEFAULT gen_random_uuid() PRIMARY KEY,
    user_id    BIGINT       NOT NULL REFERENCES users(id),
    order_no   TEXT         NOT NULL UNIQUE,
    amount     NUMERIC(12,2) NOT NULL DEFAULT 0.00,
    status     SMALLINT     NOT NULL DEFAULT 0,
    created_at TIMESTAMPTZ  NOT NULL DEFAULT now()
);
```

UUID 主键的利弊:
  优点: 分布式友好、无需协调、合并数据无冲突、不泄露业务量
  缺点:
    - 16 字节 vs BIGINT 的 8 字节，索引更大
    - 随机 UUID (v4) 导致 B-tree 索引页分裂严重，写入性能下降
    - 建议: 如果用 UUID 做主键，考虑 UUIDv7（时间排序），PostgreSQL 17+ 内置 uuidv7()
    - 或者用 UUID 做业务标识 + BIGINT 做内部主键（两列方案）

13 之前需要扩展:
CREATE EXTENSION IF NOT EXISTS "pgcrypto";       -- 提供 gen_random_uuid()
CREATE EXTENSION IF NOT EXISTS "uuid-ossp";      -- 提供 uuid_generate_v4() 等

## GENERATED COLUMNS（生成列，12+）

```sql
CREATE TABLE products (
    id         BIGINT       GENERATED ALWAYS AS IDENTITY PRIMARY KEY,
    name       TEXT         NOT NULL,
    price      NUMERIC(10,2) NOT NULL CHECK (price >= 0),
    quantity   INTEGER      NOT NULL DEFAULT 0 CHECK (quantity >= 0),
    -- STORED 生成列: 物理存储，写入时计算，可以建索引
    total_value NUMERIC(12,2) GENERATED ALWAYS AS (price * quantity) STORED
    -- PostgreSQL 18 之前只支持 STORED，不支持 VIRTUAL
    -- PostgreSQL 18 (2025-09): 终于支持 VIRTUAL 生成列！
    --   VIRTUAL 列不占磁盘空间，查询时实时计算
    --   至此 PostgreSQL 与 MySQL 一样同时支持 STORED 和 VIRTUAL 两种生成列
    -- STORED 会占用磁盘空间，但查询时不需要计算
);
```

## UNLOGGED 表（非日志表，9.1+）

```sql
CREATE UNLOGGED TABLE session_cache (
    session_id  TEXT        PRIMARY KEY,
    user_id     BIGINT      NOT NULL,
    data        JSONB,
    expires_at  TIMESTAMPTZ NOT NULL
);
```

UNLOGGED 表的特点:
  - 不写 WAL（Write-Ahead Log），写入速度快 2-10 倍
  - 崩溃后数据丢失（会被自动清空，不是损坏）
  - 不会复制到 standby 节点
  - 适用场景: 缓存、ETL 中间表、可重建的临时数据
  - 不适用: 任何不能丢失的业务数据
和临时表的区别:
  TEMPORARY TABLE: 会话级别，会话结束自动删除，其他会话看不到
  UNLOGGED TABLE: 持久存在，所有会话可见，但崩溃后清空

## 表继承 vs 分区

表继承（古老特性，不推荐用于分区目的）
PostgreSQL 独有的面向对象特性，子表继承父表的列定义
```sql
CREATE TABLE audit_log (
    id         BIGINT       GENERATED ALWAYS AS IDENTITY,
    action     TEXT         NOT NULL,
    detail     JSONB,
    created_at TIMESTAMPTZ  NOT NULL DEFAULT now()
);
```

继承主要问题: 查询父表不会使用子表的索引，约束不跨表，UNIQUE 不跨表
现代 PostgreSQL 应该用声明式分区（10+）

声明式分区（10+，推荐）
```sql
CREATE TABLE access_logs (
    id         BIGINT       GENERATED ALWAYS AS IDENTITY,
    user_id    BIGINT       NOT NULL,
    action     TEXT         NOT NULL,
    created_at TIMESTAMPTZ  NOT NULL DEFAULT now()
) PARTITION BY RANGE (created_at);
-- 注意: 分区表的主键/唯一约束必须包含分区键

CREATE TABLE access_logs_2025q1 PARTITION OF access_logs
    FOR VALUES FROM ('2025-01-01') TO ('2025-04-01');
CREATE TABLE access_logs_2025q2 PARTITION OF access_logs
    FOR VALUES FROM ('2025-04-01') TO ('2025-07-01');
CREATE TABLE access_logs_default PARTITION OF access_logs DEFAULT;
```

DEFAULT 分区（11+）: 接收不匹配任何分区的数据，强烈建议创建，否则插入不匹配的数据会报错

分区类型: RANGE（按范围）、LIST（按值列表）、HASH（按哈希，11+）
11+: 支持 DEFAULT 分区、支持更新分区键（行自动移动到正确分区）
12+: 性能大幅提升，支持外键引用分区表
13+: 支持逻辑复制分区表、支持 BEFORE INSERT 触发器

## Tablespace（表空间）

把表放在不同的物理磁盘上
```sql
CREATE TABLESPACE fast_ssd LOCATION '/mnt/ssd/pgdata';
CREATE TABLE hot_data (...) TABLESPACE fast_ssd;
```

实际使用不多，大多数场景用默认 tablespace 就够了
主要场景: SSD 放热数据，HDD 放冷数据/归档

## 其他建表技巧

CTAS（从查询建表）
```sql
CREATE TABLE active_users AS
SELECT id, username, email, created_at FROM users WHERE age >= 18;
```

> **注意**: 不复制约束、索引、主键、默认值，只复制列类型和数据

LIKE（复制表结构）
```sql
CREATE TABLE users_backup (LIKE users INCLUDING ALL);
```

INCLUDING ALL = INCLUDING CONSTRAINTS INCLUDING DEFAULTS INCLUDING INDEXES
                INCLUDING IDENTITY INCLUDING GENERATED INCLUDING STATISTICS
也可以选择性: INCLUDING DEFAULTS INCLUDING INDEXES（不含约束）

IF NOT EXISTS
```sql
CREATE TABLE IF NOT EXISTS config (
    key   TEXT PRIMARY KEY,
    value JSONB NOT NULL
);
```

## 常见错误和陷阱

1. 用 TIMESTAMP 而非 TIMESTAMPTZ
   → 永远用 TIMESTAMPTZ，除非你确切知道自己在做什么

2. 用 SERIAL 而非 IDENTITY
   → 新项目一律用 GENERATED ALWAYS AS IDENTITY

3. 用 CHAR(n)
   → PostgreSQL 中 CHAR(n) 会填充空格到指定长度，浪费空间且比较语义奇怪
   → 永远不要用 CHAR(n)，用 TEXT

4. 不加 CHECK 约束
   → PostgreSQL 的 CHECK 从第一个版本就完美支持，不加白不加

5. 忘记给外键列建索引
   → PostgreSQL 不会自动给外键列建索引（MySQL InnoDB 会）
   → DELETE/UPDATE 父表时会全表扫描子表，性能灾难
```sql
   CREATE INDEX idx_orders_user_id ON orders(user_id);
```

6. 使用表继承做分区
   → 10+ 请用声明式分区，表继承无法利用分区裁剪优化

## PostgreSQL DDL 事务性

PostgreSQL 的 DDL（CREATE TABLE、ALTER TABLE、DROP TABLE）是事务性的！
这意味着你可以:
```sql
BEGIN;
CREATE TABLE test_table (id INT);
INSERT INTO test_table VALUES (1);
ROLLBACK;  -- test_table 不会被创建
```

这是 PostgreSQL 相对于 MySQL 的重大优势（MySQL DDL 隐式 COMMIT）

## 横向对比: PostgreSQL vs 其他方言的 CREATE TABLE

### 自增策略对比

  PostgreSQL: SERIAL（语法糖，不推荐）-> IDENTITY（10+，SQL 标准，推荐）
  MySQL:      AUTO_INCREMENT（简单直接，5.7- 重启可能回退，8.0 持久化）
  Oracle:     SEQUENCE + 触发器（传统）-> IDENTITY（12c+）
              Oracle 的 SEQUENCE 是最早的实现（8i+），其他数据库的 SEQUENCE 基本参考 Oracle
  SQL Server: IDENTITY（从一开始就有）-> SEQUENCE（2012+）
              IDENTITY vs SEQUENCE 的权衡:
                IDENTITY 绑定到表列，不能跨表共享，不能在非 INSERT 场景使用
                SEQUENCE 独立对象，可跨表共享，可手动获取下一个值
                大多数 SQL Server 项目仍使用 IDENTITY（简单够用）
  SQLite:     INTEGER PRIMARY KEY 自动成为 rowid；AUTOINCREMENT 只防止 id 复用
  TiDB:       AUTO_INCREMENT（单机语义）-> AUTO_RANDOM（分布式推荐）

### 数值类型对比

  PostgreSQL: INTEGER(4B) / BIGINT(8B) / SMALLINT(2B) / NUMERIC(p,s)
              类型选择细粒度，NUMERIC 用于精确计算
  Oracle:     NUMBER 是唯一的数值类型（Oracle 独有行为）
              NUMBER(10,0) = 整数，NUMBER(10,2) = 定点数，NUMBER 无参数 = 任意精度
              没有 INT/BIGINT 等独立类型（INT 只是 NUMBER(38) 的别名）
              迁移到 PostgreSQL: NUMBER(10,0) -> BIGINT, NUMBER(10,2) -> NUMERIC(10,2)
  SQL Server: INT(4B) / BIGINT(8B) / DECIMAL(p,s) / MONEY
              MONEY 类型只有 4 位小数精度，不推荐用于需要灵活精度的场景
  MySQL:      INT(4B) / BIGINT(8B) / DECIMAL(p,s)，与 PostgreSQL 最接近

### '' = NULL（Oracle 独有行为，建表时必须考虑）

  Oracle:     空字符串 '' 等于 NULL！这是 Oracle 最大的坑
              VARCHAR2 列即使有 NOT NULL 约束，INSERT '' 也会失败
              WHERE column = '' 永远不返回行（因为 '' 是 NULL，NULL = NULL 为 UNKNOWN）
              迁移到 PostgreSQL 时: 所有 '' 检查需要改为 IS NULL 或保持 '' 语义
  PostgreSQL: '' 是空字符串，与 NULL 完全不同
  SQL Server: '' 是空字符串，与 NULL 完全不同
  MySQL:      '' 是空字符串，与 NULL 完全不同

### TEXT vs VARCHAR 对比

  PostgreSQL: TEXT 和 VARCHAR 性能完全一样，官方文档明确说"没有差异"，推荐 TEXT
  MySQL:      TEXT 存储在行外（overflow page），不能做索引前缀以外的完整索引，
              VARCHAR 存储在行内，性能更好；VARCHAR(255) 是常见选择
  Oracle:     VARCHAR2(n) 最大 4000 字节（标准模式），CLOB 用于大文本
  SQL Server: VARCHAR(MAX) 替代 TEXT（TEXT 已废弃），NVARCHAR 用于 Unicode

### DDL 事务性对比

  PostgreSQL: DDL 完全事务性！BEGIN; CREATE TABLE ...; ROLLBACK; 可以回滚建表
  MySQL:      DDL 隐式提交（CREATE TABLE 会自动 COMMIT 当前事务，无法回滚）
  Oracle:     DDL 隐式提交（同 MySQL），这是 Oracle 迁移到 PostgreSQL 时的重大行为差异
  SQL Server: DDL 事务性（同 PostgreSQL，可以回滚）
  SQLite:     DDL 事务性（同 PostgreSQL）

### 聚集索引概念（SQL Server 独有）

  SQL Server: 每张表有且仅有一个聚集索引（Clustered Index）
              聚集索引 = 表的物理存储顺序（数据按聚集索引键排列在磁盘上）
              默认主键就是聚集索引，选择不当会严重影响性能
              非聚集索引的叶节点存储聚集索引键（不是行指针 RID）
  PostgreSQL: 没有聚集索引概念（CLUSTER 命令只做一次性物理重排，不自动维护）
              所有索引指向堆表中的 ctid（物理位置指针）
  Oracle:     IOT（Index-Organized Table）类似聚集索引，但需要显式 ORGANIZATION INDEX
              默认是堆表（Heap-Organized Table）
  MySQL:      InnoDB 主键就是聚集索引（和 SQL Server 类似），存储引擎决定

### 类型严格度对比

  PostgreSQL: 最严格，需要显式 CAST（'123'::INT），不允许隐式跨类型比较
  MySQL:      宽松，大量隐式转换（'123' + 0 = 123，WHERE int_col = '123' 自动转换）
  Oracle:     中等，TO_NUMBER/TO_CHAR 显式转换为主
  SQL Server: 中等，有隐式转换规则但比 MySQL 严格
  SQLite:     极度宽松（动态类型，任何列可以存任何值，除非 STRICT 模式）

### 时间类型对比

  PostgreSQL: TIMESTAMPTZ（推荐，存 UTC 显示按 session 时区）vs TIMESTAMP（不带时区）
              两者都是 8 字节，性能无差异
  MySQL:      TIMESTAMP（4 字节，有 2038 年问题！）vs DATETIME（5 字节，无时区）
              ON UPDATE CURRENT_TIMESTAMP 自动更新（PostgreSQL 需要触发器）
  Oracle:     TIMESTAMP WITH TIME ZONE（13 字节）vs TIMESTAMP（11 字节）
  SQL Server: DATETIMEOFFSET（时区）vs DATETIME2（无时区，推荐替代 DATETIME）

### 分区对比

  PostgreSQL: 声明式分区（10+），支持 RANGE/LIST/HASH，DEFAULT 分区（11+）
  MySQL:      PARTITION BY RANGE/LIST/HASH/KEY，分区键必须在主键中（最大限制）
  Oracle:     分区功能最强大（Composite, Interval, Reference 分区等），但需要 Enterprise Edition
  SQL Server: 需要创建 PARTITION FUNCTION + PARTITION SCHEME（步骤最多）

### CHECK 约束对比

  PostgreSQL: 从第一个版本就完美支持 CHECK，最成熟
  MySQL:      5.7 及之前解析 CHECK 但不执行！8.0.16+ 才真正生效（著名的坑）
  Oracle:     完整支持 CHECK
  SQL Server: 完整支持 CHECK

### 生成列对比

  PostgreSQL: 12+ 支持 STORED 生成列；18+ 支持 VIRTUAL 生成列（里程碑式更新!）
  MySQL:      5.7+ 支持 STORED 和 VIRTUAL 两种（VIRTUAL 不占磁盘，查询时计算）
  Oracle:     11g+ 支持 VIRTUAL 列（Oracle 最早引入虚拟列）
  SQL Server: 计算列（PERSISTED = STORED，不加 PERSISTED = VIRTUAL）
