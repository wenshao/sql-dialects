## SEQUENCE: 独立的序列对象

```sql
CREATE SEQUENCE user_id_seq;

CREATE SEQUENCE order_id_seq
    AS BIGINT                    -- 数据类型（决定取值范围）
    START WITH 1
    INCREMENT BY 1
    NO MINVALUE NO MAXVALUE
    CACHE 20                     -- 每个后端预取 20 个值
    NO CYCLE;
```

使用序列
```sql
SELECT nextval('user_id_seq');             -- 获取下一个值
SELECT currval('user_id_seq');             -- 当前会话最近的 nextval
SELECT lastval();                          -- 当前会话最近任意序列的值
SELECT setval('user_id_seq', 1000);        -- 设置当前值
SELECT setval('user_id_seq', 1000, false); -- 下次 nextval 返回 1000

-- 修改序列
ALTER SEQUENCE user_id_seq RESTART WITH 1000;
ALTER SEQUENCE user_id_seq OWNED BY users.id; -- 关联到列（DROP 表自动删除）
```

## SERIAL: 语法糖（传统方式）

```sql
CREATE TABLE users (
    id BIGSERIAL PRIMARY KEY  -- 等价于 CREATE SEQUENCE + DEFAULT nextval
);
```

SERIAL 展开后的等价操作:
```sql
  CREATE SEQUENCE users_id_seq AS BIGINT;
  ALTER TABLE users ALTER COLUMN id SET DEFAULT nextval('users_id_seq');
  ALTER SEQUENCE users_id_seq OWNED BY users.id;
```

SERIAL 的问题:
  (a) 序列不归列"所有"，pg_dump 导出顺序可能导致问题
  (b) INSERT 时可以随意指定值，不会报错（缺乏保护）
  (c) 不是 SQL 标准，移植性差

## GENERATED AS IDENTITY (10+, 推荐)

GENERATED ALWAYS: 禁止手动指定值（严格模式）
```sql
CREATE TABLE orders (
    id BIGINT GENERATED ALWAYS AS IDENTITY PRIMARY KEY
);
```

GENERATED BY DEFAULT: 允许手动指定值（宽松模式）
```sql
CREATE TABLE products (
    id BIGINT GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY
);
```

带序列参数
```sql
CREATE TABLE logs (
    id BIGINT GENERATED ALWAYS AS IDENTITY
        (START WITH 1 INCREMENT BY 1 CACHE 100)
);
```

覆盖 GENERATED ALWAYS（必须显式声明意图）
```sql
INSERT INTO orders (id) OVERRIDING SYSTEM VALUE VALUES (999);
```

## 设计决策分析: SERIAL vs IDENTITY vs SEQUENCE

IDENTITY（10+ 推荐）:
  优点: SQL 标准（SQL:2003），ALWAYS 模式防止意外覆盖
  内部: 还是创建 SEQUENCE，但与列强绑定

SERIAL:
  优点: 语法简洁，向后兼容
  缺点: 非标准，序列所有权不清晰

独立 SEQUENCE:
  优点: 可跨表共享，最灵活
  场景: 多表共用一个 ID 空间（如分表场景）

## SEQUENCE 内部实现: 为什么序列不受事务回滚影响

序列值一旦通过 nextval() 分配，即使事务回滚也不会回退。
这是刻意的设计决策:

原理:
  SEQUENCE 有自己独立的事务语义（非 MVCC）。
  nextval() 直接更新 pg_sequence 的元组并持久化（WAL）。
  如果序列值随事务回滚，两个并发事务可能获得同一个值。

CACHE 的影响:
  CACHE 20 意味着每个 backend 预取 20 个值到本地内存。
  好处: 减少序列锁竞争（高并发写入瓶颈）
  代价: 后端崩溃会"丢失"未使用的缓存值 → 序列出现间隙
        多个后端交替 INSERT 时 ID 不保证连续递增

锁机制:
  nextval() 获取轻量级锁（不是事务锁），持有时间极短。
  但在超高并发下（10K+ TPS），序列锁仍可能成为瓶颈。
  解决方案: 增大 CACHE 值，或使用 UUID。

## UUID 生成: 分布式场景的替代方案

13+ 内置（无需扩展）
```sql
SELECT gen_random_uuid();    -- v4 UUID（随机）

-- uuid-ossp 扩展（更多版本）
CREATE EXTENSION IF NOT EXISTS "uuid-ossp";
SELECT uuid_generate_v4();   -- v4 随机
SELECT uuid_generate_v1();   -- v1 时间+MAC
```

使用 UUID 作为主键
```sql
CREATE TABLE sessions (
    id UUID DEFAULT gen_random_uuid() PRIMARY KEY,
    user_id BIGINT,
    created_at TIMESTAMPTZ DEFAULT NOW()
);
```

UUID vs SEQUENCE 的 trade-off:
  UUID:     无热点、无协调、全局唯一，但索引膨胀（16字节 vs 8字节BIGINT）
  SEQUENCE: 连续、紧凑、有序插入 B-tree 友好，但集中式单点
  UUID v7:  时间有序 UUID（2024 年 RFC 9562），结合两者优势

## 横向对比: 自增策略

### 语法对比

  PostgreSQL: SERIAL(语法糖) → IDENTITY(10+, SQL标准) → SEQUENCE(灵活)
  MySQL:      AUTO_INCREMENT（表级属性，每表一个，5.7-重启可能回退）
  Oracle:     SEQUENCE(传统,8i+) → IDENTITY(12c+)
  SQL Server: IDENTITY(seed,increment) → SEQUENCE(2012+)
  SQLite:     INTEGER PRIMARY KEY = rowid（AUTOINCREMENT 只防复用）

### 分布式场景

  TiDB:       AUTO_INCREMENT(兼容) + AUTO_RANDOM(分布式推荐，避免热点)
  BigQuery:   无自增（设计哲学: 分布式不应依赖全局序列）
  Spanner:    无自增（用 UUID 或 bit-reversed sequence）
  CockroachDB: unique_rowid()（混合时间戳+节点ID）

### 序列持久化

  PostgreSQL: 序列值通过 WAL 持久化，重启不回退
  MySQL 5.7:  AUTO_INCREMENT 存内存，重启取 MAX(id)+1（可能复用ID）
  MySQL 8.0:  AUTO_INCREMENT 持久化到 redo log

## 对引擎开发者的启示

(1) 序列的事务无关性是关键设计:
    nextval() 必须在事务外持久化，否则并发安全无法保证。

(2) CACHE 是性能关键:
    无 CACHE 时序列成为全局锁瓶颈（每次 nextval 需写 WAL）。
    CACHE N 将锁竞争降低 N 倍，代价是间隙和乱序。

(3) IDENTITY vs SERIAL 的教训:
    SERIAL 是 PostgreSQL 早期的设计，通过"宏展开"实现。
    IDENTITY 是 SQL 标准的做法，语义更清晰。
    这说明: 语法糖的便利性可能掩盖语义问题。

(4) 分布式引擎应考虑 UUID v7 作为默认推荐:
    时间有序 + 全局唯一，兼顾索引性能和分布式无协调。

## 版本演进

PostgreSQL 7.x:  SERIAL, CREATE SEQUENCE
PostgreSQL 10:   GENERATED AS IDENTITY（SQL:2003 标准）
PostgreSQL 13:   gen_random_uuid() 内置（无需 uuid-ossp）
PostgreSQL 16:   SEQUENCE 支持 LOGGED/UNLOGGED
