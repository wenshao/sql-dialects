# 自增与序列：各 SQL 方言实现全对比

行级唯一标识的自动生成——从单机 AUTO_INCREMENT 到分布式 UUID/Snowflake ID，各引擎的实现路线截然不同。

## 自增方案矩阵

| 引擎 | 自增语法 | SEQUENCE | SQL:2003 IDENTITY | 版本说明 |
|------|---------|----------|-------------------|---------|
| MySQL | `AUTO_INCREMENT` | 不支持 | 不支持 | 经典自增，单列限制 |
| PostgreSQL | `SERIAL`(旧) / `IDENTITY`(新) | 支持 | 支持 | 10+ 推荐 IDENTITY |
| Oracle | 无(旧) / `IDENTITY`(新) | 支持 | 支持 | 12c+ 支持 IDENTITY，之前依赖 SEQUENCE |
| SQL Server | `IDENTITY(seed, increment)` | 支持(2012+) | 部分 | IDENTITY 是私有语法，非 SQL 标准 IDENTITY |
| SQLite | `AUTOINCREMENT` / `INTEGER PRIMARY KEY` | 不支持 | 不支持 | `INTEGER PRIMARY KEY` 自动 rowid |
| MariaDB | `AUTO_INCREMENT` | 支持(10.3+) | 不支持 | 兼容 MySQL + 额外 SEQUENCE 引擎 |
| BigQuery | 无 | 不支持 | 不支持 | 设计哲学：分布式系统不应依赖全局序列 |
| Snowflake | `AUTOINCREMENT` / `IDENTITY` | 支持 | 部分 | 不保证连续，仅保证唯一 |
| TiDB | `AUTO_INCREMENT` / `AUTO_RANDOM` | 支持(6.4+) | 不支持 | 分布式推荐 AUTO_RANDOM |
| CockroachDB | `SERIAL` → `unique_rowid()` | 支持 | 支持 | SERIAL 映射为 unique_rowid()，非连续 |
| Spanner | 无原生自增 | 支持(位反转) | 不支持 | bit-reversed sequence 避免热点 |
| ClickHouse | 无 | 不支持 | 不支持 | 追加写引擎，无自增需求 |
| Hive | 无 | 不支持 | 不支持 | 批处理引擎 |
| Spark SQL | 无 | 不支持 | 不支持 | `monotonically_increasing_id()` 非连续 |
| MaxCompute | 无 | 不支持 | 不支持 | 批处理引擎 |
| DuckDB | `GENERATED ALWAYS AS IDENTITY` | 支持 | 支持 | 遵循 SQL 标准 |
| OceanBase | `AUTO_INCREMENT` | 支持(4.0+) | 不支持 | MySQL 兼容模式 / Oracle 兼容模式 |

## 各引擎语法详解

### MySQL: AUTO_INCREMENT

```sql
CREATE TABLE users (
    id BIGINT NOT NULL AUTO_INCREMENT,
    name VARCHAR(100),
    PRIMARY KEY (id)
);

-- 获取最后插入的 ID
SELECT LAST_INSERT_ID();

-- 重置自增值
ALTER TABLE users AUTO_INCREMENT = 1000;

-- 注意: 每表只允许一个 AUTO_INCREMENT 列，且必须是索引的一部分
```

### PostgreSQL: SERIAL（旧）→ IDENTITY（新）

```sql
-- 旧方式: SERIAL（本质是创建 SEQUENCE + DEFAULT）
CREATE TABLE users (
    id SERIAL PRIMARY KEY,    -- 等价于下方三步
    name VARCHAR(100)
);
-- SERIAL 展开后等价于:
-- CREATE SEQUENCE users_id_seq;
-- id INT NOT NULL DEFAULT nextval('users_id_seq');
-- ALTER SEQUENCE users_id_seq OWNED BY users.id;

-- 新方式: IDENTITY（PostgreSQL 10+，推荐）
CREATE TABLE users (
    id INT GENERATED ALWAYS AS IDENTITY PRIMARY KEY,
    name VARCHAR(100)
);

-- GENERATED BY DEFAULT 允许手动指定值
CREATE TABLE users (
    id INT GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    name VARCHAR(100)
);

-- IDENTITY 的优势:
-- 1. 不会意外被 DROP（SERIAL 的 SEQUENCE 是独立对象）
-- 2. pg_dump/pg_restore 行为更正确
-- 3. 符合 SQL:2003 标准
-- 4. 权限管理更简单（无需额外的 SEQUENCE 权限）
```

### Oracle: SEQUENCE（旧）→ IDENTITY（新）

```sql
-- 传统方式: 使用 SEQUENCE + 触发器
CREATE SEQUENCE user_seq START WITH 1 INCREMENT BY 1 NOCACHE;

-- 手动引用
INSERT INTO users (id, name) VALUES (user_seq.NEXTVAL, 'Alice');

-- 用触发器自动填充
CREATE OR REPLACE TRIGGER trg_users_id
    BEFORE INSERT ON users
    FOR EACH ROW
BEGIN
    :NEW.id := user_seq.NEXTVAL;
END;

-- Oracle 12c+: IDENTITY 列
CREATE TABLE users (
    id NUMBER GENERATED ALWAYS AS IDENTITY,
    name VARCHAR2(100)
);

-- GENERATED BY DEFAULT 允许手动指定
CREATE TABLE users (
    id NUMBER GENERATED BY DEFAULT AS IDENTITY (START WITH 1 INCREMENT BY 1),
    name VARCHAR2(100)
);

-- GENERATED BY DEFAULT ON NULL: 仅当插入值为 NULL 时自动生成
CREATE TABLE users (
    id NUMBER GENERATED BY DEFAULT ON NULL AS IDENTITY,
    name VARCHAR2(100)
);
```

### SQL Server: IDENTITY + SEQUENCE

```sql
-- IDENTITY(seed, increment)
CREATE TABLE users (
    id INT IDENTITY(1, 1) PRIMARY KEY,
    name NVARCHAR(100)
);

-- 获取最后插入的 IDENTITY 值
SELECT SCOPE_IDENTITY();       -- 当前作用域（推荐）
SELECT @@IDENTITY;             -- 全局（受触发器影响，不推荐）
SELECT IDENT_CURRENT('users'); -- 指定表（不限作用域）

-- 允许手动插入 IDENTITY 列
SET IDENTITY_INSERT users ON;
INSERT INTO users (id, name) VALUES (999, 'Manual');
SET IDENTITY_INSERT users OFF;

-- SEQUENCE (SQL Server 2012+)
CREATE SEQUENCE order_seq
    START WITH 1 INCREMENT BY 1
    MINVALUE 1 MAXVALUE 9999999
    CACHE 50;

SELECT NEXT VALUE FOR order_seq;
```

### SQLite: rowid 与 AUTOINCREMENT

```sql
-- 方式 1: INTEGER PRIMARY KEY（推荐）
-- 自动映射到 rowid，未指定时自动分配
CREATE TABLE users (
    id INTEGER PRIMARY KEY,
    name TEXT
);
-- 分配策略: max(rowid) + 1，删除后 ID 可复用

-- 方式 2: AUTOINCREMENT（严格单调递增）
CREATE TABLE users (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    name TEXT
);
-- 分配策略: max(曾经使用过的最大 rowid) + 1，ID 不复用
-- 使用 sqlite_sequence 内部表记录历史最大值
-- 注意: 有性能开销，SQLite 官方不推荐在多数场景使用
```

### TiDB: AUTO_INCREMENT + AUTO_RANDOM

```sql
-- MySQL 兼容: AUTO_INCREMENT
CREATE TABLE users (
    id BIGINT AUTO_INCREMENT PRIMARY KEY,
    name VARCHAR(100)
);
-- 注意: TiDB 的 AUTO_INCREMENT 默认不保证连续
-- 每个 TiDB 节点缓存一段 ID（默认 30000）
-- 多节点插入时 ID 交错分配

-- 分布式推荐: AUTO_RANDOM
CREATE TABLE users (
    id BIGINT AUTO_RANDOM PRIMARY KEY,
    name VARCHAR(100)
);
-- AUTO_RANDOM 结构: [符号位][分片位(默认5)][自增位]
-- 分片位随机生成 → 数据均匀分散到不同 Region
-- 解决热点写入问题
-- 不支持显式指定值（除非设置 @@allow_auto_random_explicit_insert）
```

### CockroachDB: unique_rowid() 与 SERIAL

```sql
-- SERIAL 映射为 unique_rowid()（非 PostgreSQL 的 SEQUENCE 语义）
CREATE TABLE users (
    id SERIAL PRIMARY KEY,   -- 实际类型是 INT8 DEFAULT unique_rowid()
    name STRING
);

-- unique_rowid() 结构: timestamp(高位) + nodeID(低位)
-- 优点: 无需跨节点协调，无锁
-- 缺点: 不连续，不从 1 开始，值很大（如 732953760862945281）

-- 如需 SQL 标准 IDENTITY
CREATE TABLE users (
    id INT GENERATED ALWAYS AS IDENTITY PRIMARY KEY,
    name STRING
);

-- UUID 方式（CockroachDB 推荐用于分布式主键）
CREATE TABLE users (
    id UUID DEFAULT gen_random_uuid() PRIMARY KEY,
    name STRING
);
```

### Spanner: 位反转序列

```sql
-- Spanner 不支持传统自增（会导致热点）
-- 使用 bit-reversed sequence（2023+）
CREATE SEQUENCE user_seq OPTIONS (sequence_kind = 'bit_reversed_positive');

CREATE TABLE users (
    id INT64 DEFAULT (GET_NEXT_SEQUENCE_VALUE(SEQUENCE user_seq)),
    name STRING(100)
) PRIMARY KEY (id);

-- 位反转策略:
-- 序列值 1 → 反转后 4611686018427387904
-- 序列值 2 → 反转后 2305843009213693952
-- 序列值 3 → 反转后 6917529027641081856
-- 高位分散 → 数据均匀分布到不同 split
-- 代价: 值不可读，不连续

-- 替代方案: UUID
CREATE TABLE users (
    id STRING(36) DEFAULT (GENERATE_UUID()),
    name STRING(100)
) PRIMARY KEY (id);
```

### Snowflake: AUTOINCREMENT / IDENTITY

```sql
-- AUTOINCREMENT 和 IDENTITY 是同义词
CREATE TABLE users (
    id INT AUTOINCREMENT START 1 INCREMENT 1,
    name VARCHAR(100)
);

-- 等价于
CREATE TABLE users (
    id INT IDENTITY(1, 1),
    name VARCHAR(100)
);

-- 注意:
-- 1. 不保证连续（并发插入、失败事务可能产生间隙）
-- 2. 不保证严格递增（不同微分区的插入可能交错）
-- 3. 适合做代理键，不适合做业务序号

-- ORDER 选项: 保证严格递增（性能更低）
CREATE TABLE users (
    id INT AUTOINCREMENT START 1 INCREMENT 1 ORDER,
    name VARCHAR(100)
);
-- NOORDER（默认）: 仅保证唯一
```

### SEQUENCE 语法对比

```sql
-- PostgreSQL
CREATE SEQUENCE order_seq
    AS BIGINT                  -- 数据类型（PG 10+）
    START WITH 1
    INCREMENT BY 1
    MINVALUE 1
    MAXVALUE 9223372036854775807
    CACHE 20                   -- 每会话缓存 20 个值
    NO CYCLE;                  -- 到达 MAXVALUE 时报错

SELECT nextval('order_seq');
SELECT currval('order_seq');   -- 当前会话最后获取的值
SELECT setval('order_seq', 1000);  -- 重置序列值

-- Oracle
CREATE SEQUENCE order_seq
    START WITH 1
    INCREMENT BY 1
    MINVALUE 1
    MAXVALUE 9999999999
    CACHE 20
    NOCYCLE
    ORDER;                     -- RAC 环境保证严格顺序（性能更低）

SELECT order_seq.NEXTVAL FROM DUAL;
SELECT order_seq.CURRVAL FROM DUAL;

-- SQL Server
CREATE SEQUENCE order_seq
    AS BIGINT
    START WITH 1
    INCREMENT BY 1
    MINVALUE 1
    MAXVALUE 9999999999
    CACHE 50
    NO CYCLE;

SELECT NEXT VALUE FOR order_seq;
-- SQL Server 支持在 DEFAULT 约束中使用 SEQUENCE
ALTER TABLE orders ADD CONSTRAINT df_id DEFAULT NEXT VALUE FOR order_seq FOR id;
```

### CACHE 与 NOCACHE 的影响

```
CACHE（如 CACHE 20）:
  - 每个会话/节点预取 20 个序列值到内存
  - 数据库崩溃或正常关闭时，缓存中未使用的值丢失
  - 造成间隙，但性能高（减少磁盘写入）

  示例: CACHE 20
  会话 A 获取 1-20，会话 B 获取 21-40
  会话 A 使用了 1, 2, 3 后断开
  值 4-20 丢失
  下一个值从 41 开始

NOCACHE:
  - 每次获取都写入磁盘
  - 减少间隙但不保证完全无间隙（回滚、并发抢号仍会产生间隙！）
  - 性能差（高并发下成为瓶颈）
  - ⚠️ 严格连续编号（发票号等）不能仅靠 NOCACHE，必须用应用层序号表 + 事务锁

实际建议:
  - OLTP 系统: CACHE 100-1000（性能优先）
  - 需要减少间隙: NOCACHE（但仍有回滚间隙）
  - 严格连续编号: 不用 SEQUENCE，用应用层序号表 + SELECT FOR UPDATE
```

## 间隙行为

### 间隙产生的场景

| 场景 | MySQL | PostgreSQL | Oracle | SQL Server | SQLite |
|------|-------|------------|--------|------------|--------|
| 事务回滚 | 产生间隙 | 产生间隙 | 产生间隙 | 产生间隙 | 产生间隙(AUTOINCREMENT) |
| INSERT 失败 | 产生间隙 | 产生间隙 | 产生间隙 | 产生间隙 | 不产生(INTEGER PK) |
| 批量插入 | 可能产生 | 可能产生 | CACHE 导致 | CACHE 导致 | 不产生 |
| 服务器重启 | 5.7:产生 / 8.0:不产生 | 不产生 | CACHE 导致 | CACHE 导致 | 不产生 |
| DELETE 后 | 不回填 | 不回填 | 不回填 | 不回填 | INTEGER PK 可复用 |
| 行级冲突 | 产生间隙 | 产生间隙 | 产生间隙 | 产生间隙 | - |

### MySQL 5.7 重启回退 bug

```
MySQL 5.7 及之前:
  AUTO_INCREMENT 的当前最大值存储在内存中
  服务器重启后，通过 SELECT MAX(id) FROM table 重建
  如果最大 ID 的行被删除了，重启后会分配已用过的 ID

  场景:
  1. 插入 id=1, 2, 3
  2. 删除 id=3
  3. 重启 MySQL
  4. 下一次插入得到 id=3（而非预期的 id=4）

MySQL 8.0 修复:
  AUTO_INCREMENT 计数器写入 redo log
  重启后从 redo log 恢复，不再回退
  这是 MySQL 8.0 的重要改进之一
```

### 保证无间隙的代价

```
如果业务必须要求无间隙（如发票号）:
  1. 不要使用 AUTO_INCREMENT / SEQUENCE
  2. 使用单独的计数器表 + 事务:

     BEGIN;
     UPDATE invoice_counter SET val = val + 1 WHERE name = 'invoice';
     SELECT val FROM invoice_counter WHERE name = 'invoice';
     INSERT INTO invoices (invoice_no, ...) VALUES (@val, ...);
     COMMIT;

  代价:
  - 计数器行成为全局热点（串行化）
  - 并发吞吐量大幅下降
  - 但保证无间隙（回滚时计数器也回滚）
```

## 并发安全

### MySQL innodb_autoinc_lock_mode

```
innodb_autoinc_lock_mode 控制 AUTO_INCREMENT 的锁策略:

模式 0 (traditional):
  - 所有 INSERT 持有表级 AUTO-INC 锁直到语句结束
  - 完全串行化，性能最差
  - 保证批量插入的 ID 连续
  - MySQL 5.0 之前的行为

模式 1 (consecutive，MySQL 5.7 默认):
  - 简单 INSERT: 使用轻量级互斥锁（立即释放）
  - 批量 INSERT (INSERT...SELECT, LOAD DATA): 持有表级锁
  - 保证单条语句内 ID 连续
  - 对 SBR(基于语句的复制) 安全

模式 2 (interleaved，MySQL 8.0 默认):
  - 所有 INSERT 使用轻量级互斥锁
  - 批量 INSERT 的 ID 可能交错
  - 性能最好
  - 仅对 RBR(基于行的复制) 安全
  - MySQL 8.0 默认改为 2（因默认复制也改为 RBR）
```

### PostgreSQL SEQUENCE 的并发行为

```
PostgreSQL 的 SEQUENCE 天生无锁:
  - nextval() 不在事务中（即使事务回滚，序列值也不会回退）
  - 多会话并发调用 nextval() 不会阻塞
  - 这是 SEQUENCE 会产生间隙的根本原因

SERIAL vs IDENTITY 的并发差异:
  - SERIAL: 底层使用独立 SEQUENCE 对象
  - IDENTITY: 底层也使用 SEQUENCE，但与列绑定
  - 并发行为相同，差异在于管理方式（权限、pg_dump 等）
```

## 分布式 ID 策略

### 策略对比

| 策略 | 实现引擎 | 唯一性 | 有序性 | 长度 | 热点风险 | 可读性 |
|------|---------|--------|--------|------|---------|--------|
| AUTO_INCREMENT | MySQL, MariaDB | 单机唯一 | 严格递增 | 8 字节 | 高（单点写入） | 高 |
| UUID v4 | 所有引擎 | 全局唯一 | **无序（⚠️ 聚集索引碎片化严重）** | 16 字节 | 无 | 低 |
| UUID v7 | PG 17+ 等 | 全局唯一 | 时间有序 | 16 字节 | 低 | 低 |
| Snowflake ID | 应用层 | 全局唯一 | 时间有序 | 8 字节 | 低 | 中 |
| AUTO_RANDOM | TiDB | 全局唯一 | 分片内递增 | 8 字节 | 无 | 低 |
| unique_rowid() | CockroachDB | 全局唯一 | 大致有序 | 8 字节 | 低 | 低 |
| 位反转序列 | Spanner | 全局唯一 | 无序 | 8 字节 | 无 | 低 |
| SEQUENCE + 节点偏移 | Oracle RAC | 全局唯一 | 节点内递增 | 8 字节 | 低 | 中 |

### Snowflake ID 结构（应用层方案）

```
64 位 Snowflake ID 结构:
┌─────────────────────────────────────────────────────────────────┐
│ 0 │ timestamp (41 bits)  │ machine_id (10 bits) │ seq (12 bits) │
└─────────────────────────────────────────────────────────────────┘

- 符号位: 1 bit（固定为 0）
- 时间戳: 41 bits（毫秒级，约 69 年）
- 机器 ID: 10 bits（最多 1024 个节点）
- 序列号: 12 bits（每毫秒每节点 4096 个 ID）

优点: 时间有序、全局唯一、无需协调
缺点: 依赖时钟同步、需要应用层实现

Twitter 原版使用自定义纪元 (2010-11-04)
百度 UidGenerator / 美团 Leaf 是变种实现
```

### TiDB AUTO_RANDOM 深入

```sql
-- AUTO_RANDOM 结构
CREATE TABLE t (
    id BIGINT AUTO_RANDOM(5) PRIMARY KEY,  -- 5 是分片位数
    data VARCHAR(100)
);

-- 64 位 AUTO_RANDOM 结构:
-- [符号位 1bit] [分片位 5bit] [自增位 58bit]
-- 分片位随机 → 数据均匀分布到 2^5 = 32 个 Region

-- 为什么不用 AUTO_INCREMENT?
-- TiDB 的存储层 (TiKV) 按 key 范围分片
-- AUTO_INCREMENT 的连续 ID → 所有写入集中在同一个 Region → 热点
-- AUTO_RANDOM 的随机高位 → 写入分散到多个 Region → 无热点
```

### CockroachDB unique_rowid() 深入

```sql
-- unique_rowid() = (timestamp << 15) | (nodeID % 32768)
-- 时间戳精度: 微秒
-- 节点 ID: 取模 32768

SELECT unique_rowid();
-- 返回类似: 732953760862945281

-- 优点:
-- 1. 无需跨节点协调（每个节点独立生成）
-- 2. 大致按时间有序
-- 3. 不会与其他节点冲突

-- CockroachDB 推荐的主键策略优先级:
-- 1. UUID（最均匀，推荐）
-- 2. unique_rowid()（如需大致有序）
-- 3. SEQUENCE（如需连续，但有性能代价）
```

## SQL 标准 IDENTITY

### SQL:2003 GENERATED ALWAYS vs GENERATED BY DEFAULT

```sql
-- GENERATED ALWAYS: 禁止手动指定值
CREATE TABLE t (
    id INT GENERATED ALWAYS AS IDENTITY
);
INSERT INTO t (id) VALUES (100);  -- 报错！不允许手动赋值

-- GENERATED BY DEFAULT: 允许手动指定值（未指定时自动生成）
CREATE TABLE t (
    id INT GENERATED BY DEFAULT AS IDENTITY
);
INSERT INTO t (id) VALUES (100);  -- 成功
INSERT INTO t DEFAULT VALUES;     -- 自动生成
```

### 各引擎 SQL:2003 IDENTITY 支持度

| 引擎 | GENERATED ALWAYS | GENERATED BY DEFAULT | ON NULL | IDENTITY 选项 |
|------|-----------------|---------------------|---------|--------------|
| PostgreSQL 10+ | 支持 | 支持 | 不支持 | START, INCREMENT, MINVALUE, MAXVALUE, CACHE, CYCLE |
| Oracle 12c+ | 支持 | 支持 | 支持 | 完整 SEQUENCE 选项 |
| DB2 | 支持 | 支持 | 不支持 | 完整支持（SQL 标准先驱） |
| CockroachDB | 支持 | 支持 | 不支持 | 兼容 PostgreSQL |
| DuckDB | 支持 | 支持 | 不支持 | 基本选项 |
| H2 | 支持 | 支持 | 不支持 | 基本选项 |
| SQL Server | 不支持 | 不支持 | 不支持 | 使用私有 `IDENTITY(seed, inc)` 语法 |
| MySQL | 不支持 | 不支持 | 不支持 | 使用私有 `AUTO_INCREMENT` 语法 |

```
Oracle 独有的 GENERATED BY DEFAULT ON NULL:
  当 INSERT 显式传入 NULL 时也自动生成值

  -- GENERATED BY DEFAULT（无 ON NULL）:
  INSERT INTO t (id) VALUES (NULL);  -- 插入 NULL（可能违反 NOT NULL 约束）

  -- GENERATED BY DEFAULT ON NULL:
  INSERT INTO t (id) VALUES (NULL);  -- 自动生成值（NULL 被替换）

  这在数据迁移和 ETL 场景中非常有用:
  源系统可能传入 NULL，ON NULL 保证总能生成有效值
```

## 横向总结

### 按场景选择策略

| 场景 | 推荐方案 | 不推荐方案 | 原因 |
|------|---------|-----------|------|
| 单机 OLTP | AUTO_INCREMENT / IDENTITY | UUID | 自增 ID 索引紧凑，写入有序 |
| 分布式数据库 | AUTO_RANDOM / UUID / unique_rowid() | AUTO_INCREMENT | 自增 ID 导致写入热点 |
| 业务单号(发票等) | 计数器表 + 事务 | 任何自增/SEQUENCE | 业务要求无间隙 |
| 分库分表 | Snowflake ID / SEQUENCE 偏移 | AUTO_INCREMENT | 跨库唯一性 |
| 时序数据 | 时间戳 + 机器ID | AUTO_INCREMENT | 自然有序，无协调 |
| 数据仓库/批处理 | 不需要自增 | AUTO_INCREMENT | 批处理引擎不支持，也不需要 |
| 高并发写入 | AUTO_RANDOM(TiDB) / UUID(CockroachDB) | SEQUENCE NOCACHE | 全局计数器成为瓶颈 |

### 对引擎开发者的建议

1. **自增 vs SEQUENCE 的设计选择**

```
自增（AUTO_INCREMENT / IDENTITY）:
  - 绑定在列定义中，与表生命周期一致
  - 实现简单: 原子计数器 + WAL 持久化
  - 局限: 不能跨表共享，不能独立管理

SEQUENCE（独立对象）:
  - 跨表共享，独立于表的生命周期
  - 实现复杂: 需要 DDL 支持、权限管理、持久化
  - 灵活: 可自定义 CACHE/CYCLE/MINVALUE/MAXVALUE
  - 推荐: 如果从零实现 SQL 引擎，优先支持 SQL:2003 IDENTITY
    （底层可用 SEQUENCE 实现，但对用户暴露标准语法）
```

2. **CACHE 的实现要点**

```
SEQUENCE CACHE 的核心是预分配:
  1. 节点 A 请求: 从持久化存储获取 [1, 20] 区间
  2. 持久化存储记录: 下一个可分配的起点 = 21
  3. 节点 A 在内存中依次分配 1, 2, 3, ...
  4. 节点 A 崩溃: 4-20 丢失（可接受的间隙）
  5. 节点 B 请求: 获取 [21, 40]

  关键决策:
  - CACHE 大小: 越大性能越好，间隙风险越大
  - 持久化方式: WAL / 系统表 / etcd(分布式)
  - RAC/集群: 每个节点独立缓存 vs 全局协调
```

3. **分布式场景的 ID 设计**

```
分布式 SQL 引擎在设计主键策略时的核心权衡:

  连续性 ←→ 分散性
  ↑              ↑
  对人类友好      对存储引擎友好
  索引紧凑        避免热点

  Spanner 的答案: 位反转（牺牲可读性换取分散性）
  TiDB 的答案: AUTO_RANDOM（牺牲连续性换取分散性）
  CockroachDB 的答案: unique_rowid()（牺牲连续性换取无协调）
  BigQuery 的答案: 不提供自增（从设计层面拒绝全局序列）
```
