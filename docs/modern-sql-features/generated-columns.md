# GENERATED 计算列

由表达式自动派生的列——SQL:2003 标准的实用特性，在存储成本与计算成本之间提供选择。

## 支持矩阵

| 引擎 | STORED | VIRTUAL | 版本 | 语法关键字 |
|------|--------|---------|------|-----------|
| MySQL | 支持 | 支持 | 5.7+ | `GENERATED ALWAYS AS (...) STORED/VIRTUAL` |
| PostgreSQL | 支持 | 不支持 | 12+ | `GENERATED ALWAYS AS (...) STORED` |
| SQLite | 支持 | 支持 | 3.31+ | `GENERATED ALWAYS AS (...) STORED/VIRTUAL` |
| Oracle | 不支持 | 支持 | 11g+ | `AS (expr)` —— 虚拟列 |
| SQL Server | 支持 | 支持 | 2005+ | `AS (expr) [PERSISTED]` —— 计算列 |
| MariaDB | 支持 | 支持 | 5.2+ | `GENERATED ALWAYS AS (...) {STORED|VIRTUAL|PERSISTENT}` |
| H2 | 支持 | 支持 | 1.4+ | `GENERATED ALWAYS AS (...)` |
| CockroachDB | 支持 | 支持 | 22.1+ | `AS (expr) STORED` / `AS (expr) VIRTUAL` |
| DuckDB | 不支持 | 不支持 | - | 无原生支持 |
| ClickHouse | 支持 | 不支持 | 20.1+ | `MATERIALIZED expr` / `ALIAS expr` |
| Snowflake | 不支持 | 不支持 | - | 无原生支持（用 VIEW 替代） |
| BigQuery | 不支持 | 不支持 | - | 无原生支持 |

## 核心概念: STORED vs VIRTUAL

### STORED（物理存储 / 持久化）

- 表达式结果在 INSERT/UPDATE 时计算并写入磁盘
- 读取时直接取值，无需重新计算
- 占用额外存储空间
- 可以创建索引

```sql
-- 写入时: INSERT INTO users(first_name, last_name) VALUES ('张', '三')
-- 引擎自动计算: full_name = '张' || ' ' || '三' = '张 三'
-- 磁盘上存储: first_name='张', last_name='三', full_name='张 三'
```

### VIRTUAL（虚拟 / 查询时计算）

- 不占用存储空间（除索引外）
- 每次查询时实时计算
- 某些引擎允许在虚拟列上创建索引（如 MySQL、Oracle）
- 写入性能更好（无需计算和存储）

```sql
-- 磁盘上只存储: first_name='张', last_name='三'
-- SELECT full_name 时: 引擎实时计算 '张' || ' ' || '三'
```

### 选择建议

| 场景 | 推荐类型 | 原因 |
|------|---------|------|
| 读多写少，表达式复杂 | STORED | 避免重复计算 |
| 写多读少 | VIRTUAL | 避免写入开销 |
| 需要建索引 | STORED（更通用） | 所有引擎都支持 STORED 列索引 |
| 表达式简单（如拼接字段） | VIRTUAL | 计算代价低 |
| 磁盘空间敏感 | VIRTUAL | 零额外存储 |

## 语法对比

### MySQL 5.7+

```sql
CREATE TABLE users (
    id INT PRIMARY KEY AUTO_INCREMENT,
    first_name VARCHAR(50),
    last_name VARCHAR(50),
    -- VIRTUAL: 查询时计算（默认）
    full_name VARCHAR(101) GENERATED ALWAYS AS (CONCAT(first_name, ' ', last_name)) VIRTUAL,
    -- STORED: 写入时计算并持久化
    email_domain VARCHAR(100) GENERATED ALWAYS AS (SUBSTRING_INDEX(email, '@', -1)) STORED,
    email VARCHAR(200),
    birth_date DATE,
    -- 从 JSON 字段提取
    profile JSON,
    age INT GENERATED ALWAYS AS (TIMESTAMPDIFF(YEAR, birth_date, CURDATE())) VIRTUAL,
    city VARCHAR(100) GENERATED ALWAYS AS (JSON_UNQUOTE(JSON_EXTRACT(profile, '$.city'))) VIRTUAL
);

-- MySQL VIRTUAL 列可以创建索引（B-tree 和全文索引）
CREATE INDEX idx_full_name ON users(full_name);
-- 但 VIRTUAL 列不能在索引中作为前缀列（InnoDB 限制）
```

### PostgreSQL 12+

```sql
CREATE TABLE users (
    id SERIAL PRIMARY KEY,
    first_name VARCHAR(50),
    last_name VARCHAR(50),
    -- PostgreSQL 只支持 STORED（截至 PG 17）
    full_name VARCHAR(101) GENERATED ALWAYS AS (first_name || ' ' || last_name) STORED,
    email VARCHAR(200),
    email_domain VARCHAR(100) GENERATED ALWAYS AS (split_part(email, '@', 2)) STORED,
    -- 从 JSONB 字段提取
    profile JSONB,
    city TEXT GENERATED ALWAYS AS ((profile ->> 'city')) STORED
);

-- PostgreSQL 不支持 VIRTUAL，但可以用表达式索引替代部分场景
CREATE INDEX idx_lower_email ON users (lower(email));
-- 表达式索引是 PostgreSQL 的强项，某种程度上弥补了无 VIRTUAL 列的不足
```

### Oracle 11g+

```sql
CREATE TABLE users (
    id NUMBER GENERATED BY DEFAULT AS IDENTITY,
    first_name VARCHAR2(50),
    last_name VARCHAR2(50),
    -- Oracle 的虚拟列: 只支持 VIRTUAL（不支持 STORED）
    full_name VARCHAR2(101) AS (first_name || ' ' || last_name),
    email VARCHAR2(200),
    email_domain VARCHAR2(100) AS (SUBSTR(email, INSTR(email, '@') + 1))
);

-- Oracle 虚拟列可以创建索引
CREATE INDEX idx_full_name ON users(full_name);
-- Oracle 虚拟列可以做分区键
-- 这是 Oracle 虚拟列的杀手级功能——按计算值分区
CREATE TABLE orders (
    order_id NUMBER,
    order_date DATE,
    order_month NUMBER AS (EXTRACT(MONTH FROM order_date))
) PARTITION BY LIST (order_month) (...);
```

### SQL Server（计算列）

```sql
CREATE TABLE users (
    id INT IDENTITY(1,1) PRIMARY KEY,
    first_name NVARCHAR(50),
    last_name NVARCHAR(50),
    -- 非 PERSISTED: 类似 VIRTUAL
    full_name AS (first_name + N' ' + last_name),
    -- PERSISTED: 类似 STORED
    email NVARCHAR(200),
    email_domain AS (RIGHT(email, LEN(email) - CHARINDEX('@', email))) PERSISTED
);

-- SQL Server 计算列的索引: 必须是确定性表达式
CREATE INDEX idx_full_name ON users(full_name);
-- 注意: 非 PERSISTED 计算列也可以建索引（SQL Server 会自动持久化索引中的值）
```

### SQLite 3.31+

```sql
CREATE TABLE users (
    id INTEGER PRIMARY KEY,
    first_name TEXT,
    last_name TEXT,
    full_name TEXT GENERATED ALWAYS AS (first_name || ' ' || last_name) VIRTUAL,
    data_json TEXT,
    category TEXT GENERATED ALWAYS AS (json_extract(data_json, '$.category')) STORED
);
```

### ClickHouse

```sql
CREATE TABLE users (
    id UInt64,
    first_name String,
    last_name String,
    -- MATERIALIZED: 类似 STORED（写入时计算）
    full_name String MATERIALIZED concat(first_name, ' ', last_name),
    -- ALIAS: 类似 VIRTUAL（查询时计算）
    name_length UInt32 ALIAS length(first_name) + length(last_name)
) ENGINE = MergeTree() ORDER BY id;

-- 注意: MATERIALIZED 列不包含在 SELECT * 中
-- 需要显式查询: SELECT *, full_name FROM users
```

## 实际用例

### 用例 1: 全名列

```sql
-- MySQL
ALTER TABLE employees
ADD COLUMN full_name VARCHAR(200)
    GENERATED ALWAYS AS (CONCAT_WS(' ', first_name, middle_name, last_name)) VIRTUAL;
```

### 用例 2: JSON 字段提取并索引

```sql
-- MySQL: 在 JSON 字段上创建函数索引的最佳实践
ALTER TABLE products
ADD COLUMN brand VARCHAR(100)
    GENERATED ALWAYS AS (JSON_UNQUOTE(JSON_EXTRACT(attrs, '$.brand'))) VIRTUAL;
CREATE INDEX idx_brand ON products(brand);

-- 等效于在 JSON 路径上建索引，但语义更清晰
```

### 用例 3: 哈希分片键

```sql
-- PostgreSQL: 计算哈希值用于分片
CREATE TABLE orders (
    order_id BIGINT PRIMARY KEY,
    customer_id BIGINT,
    shard_key INT GENERATED ALWAYS AS (abs(hashtext(customer_id::text)) % 16) STORED
);
```

### 用例 4: 地理坐标格式转换

```sql
-- SQL Server: 从经纬度计算 geohash
CREATE TABLE locations (
    id INT IDENTITY PRIMARY KEY,
    lat DECIMAL(10,7),
    lng DECIMAL(10,7),
    -- PERSISTED 确保只计算一次
    lat_bucket AS (CAST(FLOOR(lat * 10) AS INT)) PERSISTED,
    lng_bucket AS (CAST(FLOOR(lng * 10) AS INT)) PERSISTED
);
CREATE INDEX idx_geo_bucket ON locations(lat_bucket, lng_bucket);
```

## 对引擎开发者的实现分析

1. STORED 列的实现

STORED 列的实现相对直接，主要修改写入路径：

```
INSERT/UPDATE 流程:
1. 用户提供非计算列的值
2. 引擎检测到表有 STORED 计算列
3. 按定义的表达式计算 STORED 列的值
4. 将所有列（含 STORED 列）一起写入存储层
```

关键约束：
- STORED 列的 INSERT 语句中不能指定值
- UPDATE 同理，不能直接更新 STORED 列
- 表达式必须是确定性的（不能用 NOW()、RANDOM() 等非确定性函数）

2. VIRTUAL 列的实现（更复杂）

VIRTUAL 列需要修改查询引擎的读取路径：

```
SELECT 流程:
1. TableScan 读取物理列
2. 检测到查询涉及 VIRTUAL 列
3. 在 Project 算子中插入表达式计算节点
4. 实时计算 VIRTUAL 列的值
```

优化器考量：
- 如果 SELECT 不涉及 VIRTUAL 列，完全跳过计算
- 如果 VIRTUAL 列有索引，WHERE 过滤可以使用索引（无需计算）
- VIRTUAL 列的表达式可以参与谓词下推

3. VIRTUAL 列的索引

在 VIRTUAL 列上建索引是最精妙的特性：

```
虽然列本身不存储，但索引中存储了计算结果。
等效于 PostgreSQL 的表达式索引，但语法更友好。

写入时:
1. INSERT → 计算 VIRTUAL 列表达式 → 更新索引（但不写列数据）

查询时:
- WHERE virtual_col = 'x'  → 走索引
- SELECT virtual_col       → 从索引回表，或实时计算
```

4. 依赖管理

计算列引入了列间依赖关系：

```
full_name = first_name || last_name

依赖图: full_name → {first_name, last_name}
```

引擎需要：
- 禁止删除被计算列引用的列（或级联删除）
- 禁止修改被引用列的类型（或级联更新表达式）
- 禁止循环依赖
- 计算列不能引用其他计算列（某些引擎允许，如 SQL Server）

5. DDL 操作

```sql
-- ALTER TABLE 添加计算列
ALTER TABLE t ADD COLUMN c INT GENERATED ALWAYS AS (a + b) STORED;
-- 对于 STORED: 需要回填所有现有行（可能很慢）
-- 对于 VIRTUAL: 立即完成（无需修改数据）

-- ALTER TABLE 删除计算列
ALTER TABLE t DROP COLUMN c;
-- 检查: 是否有其他计算列依赖 c？是否有索引？
```

6. PostgreSQL 为何不支持 VIRTUAL

PostgreSQL 社区在多个版本中讨论过 VIRTUAL 列，核心顾虑：

1. **元数据存储**: VIRTUAL 列的定义需要存储在系统表中，但 PostgreSQL 的元组格式假设每列都有物理存储
2. **表达式索引已覆盖**: `CREATE INDEX ON t ((a + b))` 提供了类似功能
3. **VIEW 可替代**: `CREATE VIEW v AS SELECT *, a + b AS c FROM t` 也能实现
4. **实现复杂度**: 需要修改存储层、执行器、pg_dump、逻辑复制等多个组件

PostgreSQL 17 仍未支持 VIRTUAL 列，但社区有活跃的补丁在开发中。

## 限制与陷阱

### 通用限制

```sql
-- 计算列表达式的限制（各引擎有差异）:
-- 1. 不能引用其他表的列
-- 2. 不能使用子查询
-- 3. 不能使用非确定性函数（STORED 列通常要求）
-- 4. 不能使用用户定义的可变函数

-- MySQL 额外限制:
-- VIRTUAL 列不能有 DEFAULT 值
-- VIRTUAL 列不能被 INSERT 直接赋值
-- BEFORE INSERT 触发器看不到 VIRTUAL 列的值

-- PostgreSQL 额外限制:
-- 不支持 VIRTUAL 列
-- 计算列不能引用其他计算列
-- 不能直接用 COPY 导入计算列
```

### 性能陷阱

```sql
-- 陷阱 1: STORED 列的写入放大
-- 表有 10 个 STORED 计算列 → 每次 INSERT 需要额外计算 10 个表达式
-- 高写入场景要慎用 STORED

-- 陷阱 2: VIRTUAL 列的查询放大
-- SELECT * FROM big_table → 所有 VIRTUAL 列都会被计算
-- 即使不需要这些列，SELECT * 也会触发计算

-- 陷阱 3: 更新依赖列触发 STORED 列重算
UPDATE users SET first_name = '李' WHERE id = 1;
-- full_name (STORED) 会被自动重新计算
-- 如果有索引，索引也需要更新
```

## 各引擎特色功能

| 特性 | MySQL | PG | Oracle | SQL Server |
|------|-------|-----|--------|------------|
| VIRTUAL 列 | 支持 | 不支持 | 支持 | 支持（非 PERSISTED） |
| STORED 列 | 支持 | 支持 | 不支持 | 支持（PERSISTED） |
| 虚拟列索引 | 支持 | N/A | 支持 | 支持 |
| 虚拟列做分区键 | 不支持 | N/A | 支持 | 不支持 |
| 引用其他计算列 | 不支持 | 不支持 | 支持 | 支持 |
| 非确定性表达式 | VIRTUAL 允许 | 不支持 | 不支持 | 不支持 |
