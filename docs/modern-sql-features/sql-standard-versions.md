# SQL 标准版本与方言采用 (SQL Standard Versions and Dialect Adoption)

SQL 标准从 1986 年首次发布到 2023 年最新版，已经经历了近 40 年的演进，但没有任何一个数据库引擎完全实现过任意一版的 Full 级别——这就是 SQL 世界最大的悖论：标准存在，但人人都讲方言。

## SQL 标准时间线

ISO/IEC 9075（SQL 标准）由 ISO/IEC JTC 1/SC 32/WG 3 维护，从最初的单一文档已经分裂成 10 多个独立的 part，覆盖核心语言、外部数据封装、XML、Java 嵌入、多维数组到属性图查询等多个领域。每隔 3-7 年发布一次主要修订版，每个版本都会引入一些新特性，同时保留对前序版本的向后兼容。

### 标准发布时间线

| 标准版本 | 年份 | ISO 编号 | 主要新增能力 |
|---------|------|---------|------------|
| SQL-86 | 1986 | ANSI X3.135-1986 | 第一个 ANSI SQL 标准（DDL/DML/事务） |
| SQL-89 | 1989 | ISO 9075:1989 | 完整性约束（PRIMARY KEY/FOREIGN KEY/CHECK） |
| SQL-92 | 1992 | ISO/IEC 9075:1992 | Entry/Intermediate/Full 三级；JOIN 显式语法；INFORMATION_SCHEMA |
| SQL:1999 | 1999 | ISO/IEC 9075:1999 | 对象关系特性、CTE、递归查询、触发器、ROLE |
| SQL:2003 | 2003 | ISO/IEC 9075:2003 | XML 支持、窗口函数、MERGE、序列、生成列 |
| SQL:2006 | 2006 | ISO/IEC 9075:2006 | XQuery 整合（Part 14） |
| SQL:2008 | 2008 | ISO/IEC 9075:2008 | TRUNCATE TABLE、INSTEAD OF triggers、FETCH FIRST、ORDER BY 扩展 |
| SQL:2011 | 2011 | ISO/IEC 9075:2011 | 时态数据 (FOR SYSTEM_TIME)、Period 类型、增强窗口函数 |
| SQL:2016 | 2016 | ISO/IEC 9075:2016 | JSON 支持、行模式匹配 (MATCH_RECOGNIZE)、多态表函数 |
| SQL:2019 | 2019 | ISO/IEC 9075:2019 | Part 15: Multidimensional Arrays (MDA) |
| SQL:2023 | 2023 | ISO/IEC 9075:2023 | Part 16: Property Graph Queries (PGQ)、JSON 增强、UNIQUE NULLS |

> 标准从草案到发布通常需要 3-5 年，从发布到主流引擎实现通常再需要 5-15 年。SQL:2011 的 FOR SYSTEM_TIME 至今（2026）仅 6 个引擎完整支持。

## 长期采用滞后的根本原因

```
标准制定流程（ISO/IEC JTC 1/SC 32/WG 3）:
  Working Draft → Committee Draft → DIS → FDIS → 正式发布
  典型周期: 5-7 年

主流引擎采用流程:
  标准发布 → 厂商评估优先级 → 实现 → Beta → GA
  典型周期: 5-15 年

  Oracle: 通常 3-5 年内实现核心新特性 (商业利益驱动)
  PostgreSQL: 3-7 年 (社区驱动，质量优先)
  MySQL: 5-15 年甚至更长 (历史包袱重)
  SQLite: 选择性实现，文档化为 "Profile"
  云数据库: 根据用户需求快速跟进 (SQL:2016 JSON 普遍 2-3 年内实现)
```

## SQL 标准各版本详解

### SQL-86 / SQL-87：第一个标准

ANSI 在 1986 年发布 X3.135-1986，1987 年 ISO 接纳为 ISO 9075:1987。这一版定义了：

- 基本的 DDL：CREATE TABLE、CREATE VIEW、DROP
- 基本的 DML：SELECT、INSERT、UPDATE、DELETE
- 简单的事务：COMMIT、ROLLBACK
- 简单的权限：GRANT、REVOKE
- 数据类型：CHARACTER、DECIMAL、INTEGER、SMALLINT、FLOAT、REAL、DOUBLE PRECISION

不包括完整性约束（无 PRIMARY KEY/FOREIGN KEY 强制语法），不包括 JOIN 显式语法（仅笛卡尔积加 WHERE）。

### SQL-89：完整性约束

ISO 9075:1989 是一个相对较小的修订版，主要补充了：

- 完整性约束语法：PRIMARY KEY、FOREIGN KEY、CHECK、UNIQUE
- 默认值：DEFAULT 子句
- INTEGER 类型相关精度规定

```sql
-- SQL-89 引入的约束语法
CREATE TABLE orders (
    id INTEGER PRIMARY KEY,
    customer_id INTEGER NOT NULL,
    amount DECIMAL(10,2) DEFAULT 0,
    CHECK (amount >= 0),
    FOREIGN KEY (customer_id) REFERENCES customers(id)
);
```

### SQL-92：里程碑版本

SQL-92（ISO/IEC 9075:1992）是 SQL 标准化历史上最重要的一版，引入了大量沿用至今的核心特性：

- **三级符合性**：Entry / Intermediate / Full（后续版本简化为单一 Core）
- **显式 JOIN 语法**：INNER JOIN、LEFT JOIN、RIGHT JOIN、FULL JOIN、CROSS JOIN、NATURAL JOIN
- **集合操作**：UNION、INTERSECT、EXCEPT
- **CASE 表达式**：CASE WHEN ... THEN ... ELSE ... END
- **CAST 表达式**：CAST(x AS type)
- **INFORMATION_SCHEMA**：标准元数据视图
- **临时表**：CREATE TEMPORARY TABLE
- **新数据类型**：DATE、TIME、TIMESTAMP、INTERVAL、BIT、BLOB
- **域类型**：CREATE DOMAIN
- **权限增强**：列级权限

```sql
-- SQL-92 引入的 JOIN 语法
SELECT o.id, c.name
FROM orders o
INNER JOIN customers c ON o.customer_id = c.id
LEFT JOIN address a ON c.id = a.customer_id;

-- INFORMATION_SCHEMA 元数据查询（SQL-92 标准）
SELECT table_name, column_name, data_type
FROM information_schema.columns
WHERE table_schema = 'public';
```

> SQL-92 至今仍是大多数引擎自称的最低符合性级别，连 MySQL/SQLite 这样的引擎也只声称对 SQL-92 Core 的部分支持。

### SQL:1999：对象关系扩展

SQL:1999（也叫 SQL3）开启了"对象关系"时代，引入了大量重要特性：

- **递归查询 (CTE)**：WITH RECURSIVE
- **CTE (公共表表达式)**：WITH ... AS (...)
- **触发器**：CREATE TRIGGER
- **用户定义类型 (UDT)**：CREATE TYPE
- **数组类型**：ARRAY 集合类型
- **OLAP 扩展**：GROUPING SETS、ROLLUP、CUBE
- **角色 (ROLE)**：CREATE ROLE
- **保存点 (SAVEPOINT)**
- **正则表达式**：SIMILAR TO
- **布尔类型**：BOOLEAN

```sql
-- SQL:1999 递归 CTE：经典员工层级查询
WITH RECURSIVE employee_tree AS (
    SELECT id, name, manager_id, 1 AS level
    FROM employees
    WHERE manager_id IS NULL
    UNION ALL
    SELECT e.id, e.name, e.manager_id, et.level + 1
    FROM employees e
    JOIN employee_tree et ON e.manager_id = et.id
)
SELECT * FROM employee_tree;

-- SQL:1999 OLAP 扩展
SELECT region, product, SUM(amount)
FROM sales
GROUP BY ROLLUP(region, product);
```

### SQL:2003：XML 与窗口函数

SQL:2003 是另一个重大里程碑，引入了：

- **窗口函数**：ROW_NUMBER()、RANK()、DENSE_RANK()、LAG()、LEAD()、OVER (PARTITION BY ... ORDER BY ...)
- **MERGE 语句**：UPSERT 标准语法
- **序列对象**：CREATE SEQUENCE、NEXT VALUE FOR
- **生成列**：GENERATED ALWAYS AS
- **Identity 列**：GENERATED BY DEFAULT AS IDENTITY
- **MULTISET 类型**
- **TABLESAMPLE 子句**
- **XML 类型** (Part 14)
- **新预定义类型**：BIGINT、Multisets

```sql
-- SQL:2003 窗口函数
SELECT
    employee_id,
    department_id,
    salary,
    RANK() OVER (PARTITION BY department_id ORDER BY salary DESC) AS dept_rank,
    salary - AVG(salary) OVER (PARTITION BY department_id) AS diff_from_avg
FROM employees;

-- SQL:2003 MERGE
MERGE INTO target t
USING source s ON t.id = s.id
WHEN MATCHED THEN UPDATE SET t.value = s.value
WHEN NOT MATCHED THEN INSERT (id, value) VALUES (s.id, s.value);
```

### SQL:2006：XQuery 整合

SQL:2006 是一个相对较小的修订版，主要工作是把 W3C 的 XQuery 1.0 整合进 SQL/XML (Part 14)。引入了：

- **XMLQUERY()**：在 SQL 中嵌入 XQuery
- **XMLEXISTS()**：判断 XPath 表达式是否匹配
- **XMLNAMESPACES**：命名空间声明
- **XMLTABLE()**：将 XML 文档投影为关系表

```sql
-- SQL:2006 XMLTABLE 示例
SELECT t.product_id, t.price
FROM orders o,
     XMLTABLE('//order/item' PASSING o.xml_data
              COLUMNS
                  product_id INTEGER PATH 'product/@id',
                  price DECIMAL(10,2) PATH 'price'
     ) t;
```

### SQL:2008：实用增量

SQL:2008 引入了多个被广泛使用的特性：

- **TRUNCATE TABLE**：标准化（之前各家方言）
- **INSTEAD OF triggers**：可视图触发器
- **FETCH FIRST n ROWS ONLY** / **OFFSET n ROWS**：标准分页语法
- **ORDER BY 扩展**：可在子查询/视图/CTE 中使用
- **MERGE 增强**：DELETE 子句、多 WHEN MATCHED
- **新函数**：DIAGNOSTICS

```sql
-- SQL:2008 标准分页（替代 LIMIT）
SELECT * FROM orders
ORDER BY created_at DESC
OFFSET 100 ROWS FETCH FIRST 10 ROWS ONLY;

-- SQL:2008 标准 TRUNCATE
TRUNCATE TABLE large_log_table;
```

### SQL:2011：时态数据

SQL:2011 引入了两类重要特性：

- **应用时态表 (Application-Time Period)**：业务时间维度
- **系统版本表 (System-Versioned Table)**：自动维护历史
- **PERIOD 类型**：时间段类型
- **FOR SYSTEM_TIME AS OF / BETWEEN**：时态查询
- **窗口函数增强**：嵌套窗口、NTH_VALUE
- **OFFSET FETCH** 增强

```sql
-- SQL:2011 系统版本表
CREATE TABLE customers (
    id INTEGER PRIMARY KEY,
    name VARCHAR(100),
    valid_from TIMESTAMP GENERATED ALWAYS AS ROW START,
    valid_to TIMESTAMP GENERATED ALWAYS AS ROW END,
    PERIOD FOR SYSTEM_TIME (valid_from, valid_to)
) WITH SYSTEM VERSIONING;

-- 查询历史数据
SELECT * FROM customers
FOR SYSTEM_TIME AS OF TIMESTAMP '2025-01-01 00:00:00'
WHERE id = 100;

-- 应用时态表
CREATE TABLE employee_history (
    id INTEGER,
    name VARCHAR(100),
    salary DECIMAL(10,2),
    valid_start DATE,
    valid_end DATE,
    PERIOD FOR employment_period (valid_start, valid_end)
);
```

### SQL:2016：JSON 与行模式

SQL:2016 是云时代之前的重要修订，引入了：

- **JSON 支持**：IS JSON、JSON_VALUE()、JSON_QUERY()、JSON_TABLE()、JSON_OBJECT()、JSON_ARRAY()
- **JSON_EXISTS()**
- **行模式匹配**：MATCH_RECOGNIZE
- **多态表函数 (PTF)**：动态结果集
- **LISTAGG()** 聚合函数标准化
- **DATE_TRUNC** / **JULIAN** / **JULIAN_DAY** 等日期函数

```sql
-- SQL:2016 JSON 查询
SELECT
    JSON_VALUE(profile, '$.email') AS email,
    JSON_QUERY(profile, '$.addresses') AS addresses,
    JSON_EXISTS(profile, '$.phone') AS has_phone
FROM users
WHERE profile IS JSON;

-- JSON_TABLE 投影
SELECT t.product_id, t.qty
FROM orders o,
     JSON_TABLE(o.items, '$[*]'
         COLUMNS (
             product_id INTEGER PATH '$.product_id',
             qty        INTEGER PATH '$.quantity'
         )) AS t;

-- SQL:2016 MATCH_RECOGNIZE：股价 V 形检测
SELECT * FROM stock_prices
MATCH_RECOGNIZE (
    PARTITION BY symbol
    ORDER BY ts
    MEASURES STRT.ts AS start_ts, BOTTOM.ts AS bottom_ts, UP.ts AS end_ts
    PATTERN (STRT DOWN+ BOTTOM UP+)
    DEFINE
        DOWN AS price < PREV(price),
        BOTTOM AS price < PREV(price),
        UP AS price > PREV(price)
);
```

### SQL:2019：多维数组 (Part 15)

SQL:2019 主要工作是新增 ISO/IEC 9075-15 (Part 15) 多维数组 (Multidimensional Arrays, MDA)，扩展了 SQL/Foundation 中的数组类型，使其支持高维数据：

- **多维数组类型**：ARRAY[2][3] 等
- **数组聚合**：ARRAY_AGG over 维度
- **数组下标运算**：增强的 [i][j][k] 访问
- **维度元数据**：CARDINALITY、TRIM_ARRAY

```sql
-- SQL:2019 多维数组示例（Part 15）
DECLARE matrix INTEGER ARRAY[3][3];
SET matrix = ARRAY[[1,2,3], [4,5,6], [7,8,9]];

-- 多维下标
SELECT matrix[2][3];   -- 返回 6

-- 元素聚合
SELECT ARRAY_AGG(value ORDER BY i, j)
FROM matrix_data;
```

> Part 15 主要面向科学计算、机器学习张量数据等场景，目前仅极少数引擎（PostgreSQL 部分扩展、SciDB 等）实际实现，主流商业引擎采用率极低。

### SQL:2023：属性图 (Part 16) 与 JSON 增强

SQL:2023 是最新版本，主要引入：

- **属性图查询 (PGQ)**：ISO/IEC 9075-16，标准化的图查询语法
- **MATCH** 子句：基于路径模式
- **JSON 增强**：JSON 数据类型 (Part 6)、IS JSON 增强、JSON 数组聚合
- **MULTISET 增强**
- **UNIQUE NULLS NOT DISTINCT**：UNIQUE 约束 NULL 处理选项
- **ANY_VALUE()**：聚合函数标准化
- **STRING_AGG()**：增强标准
- **LEAST() / GREATEST()** 标准化

```sql
-- SQL:2023 PGQ：属性图查询
SELECT *
FROM GRAPH_TABLE (
    social_graph
    MATCH (a:Person)-[r:KNOWS]->(b:Person)
    WHERE a.name = 'Alice'
    COLUMNS (
        a.name AS source,
        b.name AS target,
        r.since AS since_year
    )
);

-- SQL:2023 UNIQUE NULLS NOT DISTINCT
CREATE TABLE products (
    id INTEGER,
    sku VARCHAR(50),
    UNIQUE NULLS NOT DISTINCT (sku)  -- NULL 视为重复
);
```

## ISO/IEC 9075 标准的 Parts 结构

SQL 标准从 SQL:1999 起被拆分为多个独立的 part，每个 part 由 ISO 单独编号发布：

| Part | 全称 | 范围 |
|------|------|------|
| Part 1 | Framework (SQL/Framework) | 框架与术语 |
| Part 2 | Foundation (SQL/Foundation) | 核心 SQL 语言（最重要） |
| Part 3 | Call-Level Interface (SQL/CLI) | C 语言 API（类似 ODBC） |
| Part 4 | Persistent Stored Modules (SQL/PSM) | 存储过程语言 |
| Part 9 | Management of External Data (SQL/MED) | 外部数据封装 (FDW) |
| Part 10 | Object Language Bindings (SQL/OLB) | （已废弃，原 Java） |
| Part 11 | Information and Definition Schemas | INFORMATION_SCHEMA |
| Part 13 | Routines and Types Using Java | SQL/JRT (Java 嵌入) |
| Part 14 | XML-Related Specifications (SQL/XML) | XML 类型与 XQuery 整合 |
| Part 15 | Multi-Dimensional Arrays (SQL/MDA) | 多维数组（SQL:2019 引入） |
| Part 16 | Property Graph Queries (SQL/PGQ) | 属性图查询（SQL:2023 引入） |

> Part 5/6/7/8/12 已被合并或废弃，不再独立维护。Part 2 (Foundation) 占了标准 80% 以上的内容，是引擎实现的主要工作量来源。

## 各引擎 SQL 标准采用画像

### Oracle Database

Oracle 是商业关系数据库标杆，对标准的跟进相对积极：

- 1979 年 Oracle V2 早于 SQL-86 标准
- 8i (1999): 实现 SQL:1999 大部分特性（CTE、递归 CONNECT BY 替代、对象类型）
- 9i/10g (2001-2003): 窗口函数（早于 SQL:2003 GA）
- 11g/12c: 加入 SQL:2008 扩展，FETCH FIRST、Identity 列
- 12c (2013): SQL:2011 时态特性 (Flashback Archive)
- 19c/21c: SQL:2016 JSON 完整支持，21c 增加原生 JSON 类型
- 23ai (2024): SQL:2023 PGQ 属性图、向量类型

```sql
-- Oracle 长期独有的 CONNECT BY 递归（早于 SQL:1999）
SELECT employee_id, name, LEVEL
FROM employees
START WITH manager_id IS NULL
CONNECT BY PRIOR employee_id = manager_id;

-- 后续才加入 SQL:1999 标准的 WITH RECURSIVE
WITH employee_tree (id, name, lvl) AS (
    SELECT employee_id, name, 1 FROM employees WHERE manager_id IS NULL
    UNION ALL
    SELECT e.employee_id, e.name, et.lvl + 1
    FROM employees e JOIN employee_tree et ON e.manager_id = et.id
)
SELECT * FROM employee_tree;
```

### IBM DB2

DB2 是 SQL 标准的诞生地（IBM SystemR），对标准的支持深度无人能及：

- DB2 v8 (2004): SQL:2003 完整窗口函数
- DB2 9.5 (2007): SQL/XML
- DB2 10 (2012): SQL:2011 时态表完整实现
- DB2 11 (2016+): SQL:2016 JSON

DB2 在 SQL:2011 时态特性上的实现是业界最完整的，支持系统时间表、应用时间表和双时态表。

### Microsoft SQL Server

T-SQL 与 SQL 标准的关系最为微妙：

- SQL Server 2005: TOP 替代 LIMIT/FETCH FIRST，OUTPUT 子句（非标准）
- SQL Server 2008: MERGE 标准化
- SQL Server 2012: OFFSET FETCH 标准分页（之前是 TOP+ROW_NUMBER）
- SQL Server 2016: Temporal Tables（部分 SQL:2011）
- SQL Server 2017: GRAPH 表（自定义，非 PGQ）
- SQL Server 2022: SQL:2016 JSON 增强

```sql
-- SQL Server 自有的 GRAPH 语法（2017+，非 SQL:2023 PGQ）
CREATE TABLE Person (id INT, name VARCHAR(100)) AS NODE;
CREATE TABLE Knows AS EDGE;

SELECT p1.name, p2.name
FROM Person p1, Knows k, Person p2
WHERE MATCH(p1-(k)->p2);
```

### PostgreSQL

PostgreSQL 自称"最先进的开源关系数据库"，标准符合度最高：

- 8.4 (2009): 窗口函数 + CTE
- 9.5 (2016): UPSERT (INSERT ON CONFLICT)、TABLESAMPLE、行级安全
- 14: SQL:2016 部分 JSON
- 15: MERGE 语句标准化
- 16: SQL:2016 SQL/JSON 路径表达式、IS JSON
- 17: JSON_TABLE、JSON_VALUE、JSON_QUERY 等 SQL:2016 函数

PostgreSQL 在 SQL:2016 JSON 支持上的进度从 2022 年的 14 版本开始加速，到 17 版本基本完整。

### MySQL

MySQL 在 SQL 标准跟进上长期保守，但 8.0 后明显改善：

- 5.x 系列: 仅声称 SQL-92，无窗口函数、无 CTE
- 8.0 (2018): CTE、窗口函数（SQL:2003）、JSON 类型
- 8.4 (2024): 部分 SQL:2016 JSON 函数
- 9.0+: 向量类型、改进 JSON

MySQL 至今不支持：MERGE 语句、SQL:2011 时态表、FETCH FIRST 标准语法（用 LIMIT）、INTERSECT/EXCEPT 自 8.0.31 才支持。

### MariaDB

MariaDB 从 MySQL 5.5 分叉后在某些标准特性上比 MySQL 更激进：

- 10.2 (2016): CTE + 窗口函数
- 10.3 (2018): SQL:2011 系统版本表（开源唯一完整实现）
- 10.5+: INTERSECT/EXCEPT、Application Time

MariaDB 是少数完整支持 SQL:2011 双时态特性的开源引擎。

### SQLite

SQLite 是一个特殊存在，官方明确放弃完整符合声明：

- 维护文档化的 [Conformance Profile](https://sqlite.org/lang.html)
- 3.25 (2018): 窗口函数（追上 SQL:2003）
- 3.35 (2021): UPSERT (ON CONFLICT)
- 3.39 (2022): RIGHT/FULL JOIN（之前不支持）
- JSON1 扩展: 大部分 SQL:2016 JSON 操作

SQLite 因为嵌入式定位，不实现 MERGE、不实现存储过程（无 SQL/PSM）、不实现网络协议相关特性，但在 SQL:2003 之前的核心特性上覆盖度高。

## SQL 各 Part 的引擎采用情况

### Part 1: Framework

所有声称符合 SQL 标准的引擎都隐含支持。

### Part 2: Foundation（核心）

最主要的标准部分。所有关系引擎都不同程度实现：

| 引擎 | Foundation 符合度 |
|------|-----------------|
| Oracle | ~85% Core SQL:2003 + 多项 SQL:2008/2011/2016 选项 |
| DB2 | ~85% Core SQL:2003 + 多项 SQL:2008/2011 选项 |
| PostgreSQL | ~85% Core SQL:2016 |
| SQL Server | ~75% Core SQL:2003 + 子集 SQL:2016 |
| MySQL | ~60% Core SQL:1992 + 部分 SQL:2003 |

### Part 3: SQL/CLI

类似 ODBC 的 C 接口，实际用得不多（多数引擎用厂商驱动或 ODBC）：

| 引擎 | SQL/CLI 实现 |
|------|------------|
| ODBC（事实标准） | 几乎所有引擎通过 ODBC 桥接 |
| Oracle OCI | Oracle 自有，部分兼容 |

| PostgreSQL libpq | 自有 API，非 SQL/CLI |

### Part 4: SQL/PSM

| 引擎 | PSM 兼容度 |
|------|----------|

| PostgreSQL | PL/pgSQL（语法不同，但概念对齐） |
| Oracle | PL/SQL（早于 PSM，影响了 PSM 设计） |
| SQL Server | T-SQL（与 PSM 风格差异大） |
| Snowflake | JavaScript / Python 过程 + Snowflake Scripting (PL/SQL 风格) |

### Part 9: SQL/MED（外部数据封装）

| 引擎 | SQL/MED 实现 |
|------|------------|

| Oracle | 部分（External Tables, DB Links） |

PostgreSQL 是 SQL/MED 标准的最完整实现，支持丰富的 FDW 生态（postgres_fdw、mysql_fdw、Oracle FDW、MongoDB FDW 等）。

| Oracle 23ai | 完整 |
| Google Spanner | 部分 (GoogleSQL Graph) |
| 其他 | 不支持，使用 Cypher / Gremlin / GQL |

注意：ISO 在 2024 年单独发布了 ISO/IEC 39075:2024（GQL），与 SQL/PGQ 是不同的图查询标准，前者独立于 SQL，后者作为 SQL 的扩展。

## 标准采用滞后分析

### 滞后时间统计

| 特性 | 标准发布 | 主流引擎完整支持 | 滞后年数 |
|------|---------|----------------|---------|
| JOIN 显式语法 | 1992 | ~1995-2000 | 3-8 年 |
| 窗口函数 (Oracle) | 2003 | 2003-2009 | 0-6 年 |
| 窗口函数 (MySQL) | 2003 | 2018 | 15 年 |
| MERGE | 2003 | 2007-2015 | 4-12 年 |
| FETCH FIRST | 2008 | 2012-2018 | 4-10 年 |
| 时态表 (DB2) | 2011 | 2012 | 1 年 |
| 时态表 (PostgreSQL) | 2011 | 至今未原生 | 14+ 年 |
| JSON (PostgreSQL 17) | 2016 | 2024 | 8 年 |
| JSON (MySQL) | 2016 | 部分 (2024+) | 8+ 年 |
| Property Graph | 2023 | 2024 (Oracle 23ai) | 1 年 |

### 滞后的根本原因

1. **历史包袱**：MySQL 5.x 和 SQLite 早期专注嵌入式/Web 场景，许多企业级特性优先级低
2. **语法冲突**：旧版本的非标准语法（如 LIMIT vs FETCH FIRST）已经形成生态，强制改变破坏兼容性
3. **实现复杂度**：时态表、窗口函数、PGQ 都需要存储引擎和查询优化器深度配合
4. **商业优先级**：商业数据库根据客户需求实现新特性，而非追求标准符合度
5. **模块化标准**：Part 4 (PSM) / Part 9 (MED) / Part 14 (XML) 等被实际使用率较低
6. **开源社区资源**：开源项目可能因为开发者数量有限，长期推迟某个大特性
7. **云原生重定向**：云数据库更愿意实现自有便利特性 (Snowflake Time Travel) 而非标准 (SQL:2011 时态)

### 标准的形成方式

```
正向：标准跟随实践
  Oracle PL/SQL → SQL/PSM (Part 4)
  Oracle CONNECT BY → SQL:1999 WITH RECURSIVE
  T-SQL OUTPUT → SQL:2011 部分采纳

反向：标准引领实践
  SQL:2003 窗口函数（多家协商）→ 各家实现
  SQL:2016 JSON_TABLE → 标准制定后多家陆续实现
  SQL:2023 PGQ → Oracle 第一个实现，其他引擎跟进
```

## 关键发现

1. **没有引擎完整实现任何一版 Full**。所有引擎都只对应 Core SQL 加部分可选特性。
2. **窗口函数普及度最高**。SQL:2003 的窗口函数现在几乎是新引擎的标配，连 SQLite 都已实现。
3. **MySQL 的"双面性"**：自我声明 SQL-92 但实际有 SQL:2003 窗口函数和 SQL:2016 部分 JSON。
4. **时态表是开源软肋**。SQL:2011 至今 15 年，只有 MariaDB 在开源关系引擎中完整支持。
5. **JSON 标准化滞后于实际使用**。MySQL/PostgreSQL 早在 SQL:2016 之前就支持 JSON 类型，但语法不同于标准；PostgreSQL 17 是补完标准 JSON_TABLE/JSON_VALUE 的最新例子。
6. **PostgreSQL 是开源标准符合度王者**。从 SQL-92 到 SQL:2016 JSON，几乎所有重要特性都有原生实现。
7. **云数据库自创扩展现象普遍**：Snowflake Time Travel、BigQuery 数组、Databricks VARIANT 等绕过标准创新。
8. **SQL:1999 递归 CTE 是分水岭**。在它之前 Oracle 用 CONNECT BY，在它之后所有主流引擎都改用 WITH RECURSIVE。
9. **SQL:2011 的 FETCH FIRST 普及度低于预期**。MySQL/MariaDB/SQLite 至今用 LIMIT，原因是早期生态依赖。
10. **SQL:2023 PGQ 标准的实现速度比预期快**。Oracle 23ai 在 2024 年就发布了完整实现，远早于多数 SQL:2016 特性的采用速度。
11. **Part 9 SQL/MED 是"PostgreSQL 一家独大"**。FDW 生态独占鳌头，其他引擎几乎无对应实现。
12. **SQLite 的"诚实姿态"值得借鉴**。明确标注 Conformance Profile，比一些引擎含糊的"基本符合"更实用。
13. **方言之间的相互模仿**。MySQL 兼容引擎（TiDB/OceanBase/StarRocks/Doris/SingleStore）将 MySQL 方言事实上提升为"准标准"。
14. **PostgreSQL wire 协议成为云数据库标准**。CockroachDB/YugabyteDB/Greenplum/Materialize/RisingWave 等都通过 PostgreSQL wire 协议接入，导致 PostgreSQL SQL 方言事实上成为分析/分布式领域第二个"准标准"。
15. **Part 15 (MDA) 几乎被忽视**。SQL:2019 多维数组目前主要在科学计算领域使用，关系数据库和分析数据库基本未实现。
