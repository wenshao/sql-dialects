# MySQL → PostgreSQL 迁移指南

MySQL 到 PostgreSQL 是最常见的迁移路径之一。本文档从 SQL 语法层面列出所有需要修改的点。

## 核心差异总览

| 维度 | MySQL | PostgreSQL | 迁移影响 |
|------|-------|-----------|---------|
| 类型严格度 | 宽松（隐式转换） | 严格（需要显式 CAST） | **高**: 大量查询需要加 CAST |
| 标识符引用 | \`反引号\` | "双引号" | **高**: 所有引用标识符需替换 |
| 字符串引用 | 单引号或双引号 | 只有单引号 | **中**: 双引号在 PG 中是标识符 |
| 自增 | AUTO_INCREMENT | SERIAL / IDENTITY | **中**: DDL 需要重写 |
| DDL 事务性 | 隐式提交 | 可回滚 | **低**: PG 更安全 |
| 默认隔离 | REPEATABLE READ | READ COMMITTED | **中**: 锁行为可能不同 |

## DDL 迁移

### 数据类型映射

| MySQL 类型 | PostgreSQL 类型 | 说明 |
|-----------|----------------|------|
| `TINYINT` | `SMALLINT` | PG 无 TINYINT |
| `TINYINT(1)` / `BOOLEAN` | `BOOLEAN` | PG 有真正的布尔 |
| `TINYINT UNSIGNED` | `SMALLINT` + CHECK | PG 无 UNSIGNED |
| `INT` | `INTEGER` | 直接映射 |
| `INT UNSIGNED` | `BIGINT` | 用更大类型覆盖范围 |
| `BIGINT` | `BIGINT` | 直接映射 |
| `BIGINT UNSIGNED` | `NUMERIC(20,0)` | PG 的 BIGINT 也是有符号 |
| `FLOAT` | `REAL` | 4 字节浮点 |
| `DOUBLE` | `DOUBLE PRECISION` | 8 字节浮点 |
| `DECIMAL(M,D)` | `NUMERIC(M,D)` | 直接映射 |
| `VARCHAR(N)` | `VARCHAR(N)` 或 `TEXT` | PG 推荐 TEXT |
| `CHAR(N)` | `CHAR(N)` | 直接映射 |
| `TINYTEXT` | `TEXT` | PG 只有一种 TEXT |
| `TEXT` | `TEXT` | 直接映射 |
| `MEDIUMTEXT` | `TEXT` | PG 只有一种 TEXT |
| `LONGTEXT` | `TEXT` | PG TEXT 最大 1GB |
| `BLOB` | `BYTEA` | 不同的二进制接口 |
| `MEDIUMBLOB` / `LONGBLOB` | `BYTEA` | PG BYTEA 最大 1GB |
| `DATE` | `DATE` | 直接映射 |
| `DATETIME` | `TIMESTAMP` | PG 的 TIMESTAMP 无时区 |
| `DATETIME(6)` | `TIMESTAMP(6)` | 微秒精度 |
| `TIMESTAMP` | `TIMESTAMPTZ` | PG 的 TIMESTAMPTZ 带时区 |
| `TIME` | `TIME` | 直接映射 |
| `YEAR` | `SMALLINT` | PG 无 YEAR 类型 |
| `ENUM('a','b','c')` | `CREATE TYPE ... AS ENUM(...)` | PG 需要先建类型 |
| `SET('a','b','c')` | `TEXT[]` | 用数组替代 |
| `JSON` | `JSONB` | 推荐 JSONB（二进制存储，支持索引） |
| `BIT(N)` | `BIT(N)` | 直接映射 |
| `BINARY(N)` | `BYTEA` | PG 无定长二进制 |

### 自增主键

```sql
-- MySQL
CREATE TABLE users (
    id BIGINT NOT NULL AUTO_INCREMENT,
    name VARCHAR(64),
    PRIMARY KEY (id)
);

-- PostgreSQL 方案 1: SERIAL（旧方式）
CREATE TABLE users (
    id BIGSERIAL PRIMARY KEY,
    name VARCHAR(64)
);

-- PostgreSQL 方案 2: IDENTITY（10+，推荐）
CREATE TABLE users (
    id BIGINT GENERATED ALWAYS AS IDENTITY PRIMARY KEY,
    name VARCHAR(64)
);
```

注意差异:
- `SERIAL` 内部创建 SEQUENCE，DROP TABLE 时需要手动清理（或用 CASCADE）
- `GENERATED ALWAYS` 不允许手动指定值（用 `GENERATED BY DEFAULT` 允许）
- MySQL 的 `AUTO_INCREMENT = N` 在 PG 用 `ALTER SEQUENCE ... RESTART WITH N`

### 表选项

```sql
-- MySQL（这些在 PG 中不需要 / 不存在）
CREATE TABLE t (...)
    ENGINE=InnoDB                    -- PG: 无引擎概念
    DEFAULT CHARSET=utf8mb4          -- PG: 建库时指定编码
    COLLATE=utf8mb4_unicode_ci       -- PG: 列级或建库时指定
    AUTO_INCREMENT=1000              -- PG: ALTER SEQUENCE
    COMMENT='表注释';                -- PG: COMMENT ON TABLE t IS '...'
```

### ENUM 类型

```sql
-- MySQL: 内联定义
CREATE TABLE t (status ENUM('active', 'inactive', 'deleted'));

-- PostgreSQL: 先建类型再引用
CREATE TYPE status_enum AS ENUM ('active', 'inactive', 'deleted');
CREATE TABLE t (status status_enum);
```

### ON UPDATE CURRENT_TIMESTAMP

```sql
-- MySQL: 列级属性
updated_at DATETIME DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP

-- PostgreSQL: 需要触发器
updated_at TIMESTAMPTZ DEFAULT CURRENT_TIMESTAMP
-- + 触发器函数
CREATE FUNCTION update_timestamp() RETURNS TRIGGER AS $$
BEGIN NEW.updated_at = CURRENT_TIMESTAMP; RETURN NEW; END;
$$ LANGUAGE plpgsql;
CREATE TRIGGER t BEFORE UPDATE ON users FOR EACH ROW EXECUTE FUNCTION update_timestamp();
```

## DML 迁移

### INSERT

| MySQL | PostgreSQL | 说明 |
|-------|-----------|------|
| `INSERT ... SET col=val` | 不支持 | 用标准 INSERT INTO ... VALUES |
| `INSERT IGNORE INTO` | `INSERT ... ON CONFLICT DO NOTHING` | 需要 9.5+ |
| `REPLACE INTO` | `INSERT ... ON CONFLICT DO UPDATE` | 语义不完全等价（REPLACE=DELETE+INSERT） |
| `ON DUPLICATE KEY UPDATE` | `ON CONFLICT (col) DO UPDATE SET` | PG 需要显式指定冲突列 |
| `VALUES()` in ON DUPLICATE KEY | `EXCLUDED.*` in ON CONFLICT | 关键字不同 |
| `LAST_INSERT_ID()` | `RETURNING id` 或 `currval()` | PG 推荐 RETURNING |

### UPDATE

```sql
-- MySQL: 多表 UPDATE
UPDATE users u JOIN orders o ON u.id = o.user_id
SET u.status = 1 WHERE o.amount > 1000;

-- PostgreSQL: FROM 子句
UPDATE users SET status = 1
FROM orders WHERE users.id = orders.user_id AND orders.amount > 1000;
```

### DELETE

```sql
-- MySQL: 多表 DELETE
DELETE u FROM users u JOIN blacklist b ON u.email = b.email;

-- PostgreSQL: USING 子句
DELETE FROM users USING blacklist WHERE users.email = blacklist.email;
```

### LIMIT

```sql
-- MySQL: UPDATE/DELETE 支持 LIMIT
UPDATE users SET status = 0 WHERE status = 1 LIMIT 100;
DELETE FROM users WHERE status = 0 ORDER BY id LIMIT 100;

-- PostgreSQL: 不支持，需要子查询
UPDATE users SET status = 0 WHERE id IN (
    SELECT id FROM users WHERE status = 1 LIMIT 100
);
DELETE FROM users WHERE id IN (
    SELECT id FROM users WHERE status = 0 ORDER BY id LIMIT 100
);
```

## 查询语法迁移

### 字符串

| MySQL | PostgreSQL | 说明 |
|-------|-----------|------|
| `CONCAT(a, b)` | `a \|\| b` 或 `CONCAT(a, b)` | PG 的 `\|\|` 是拼接运算符 |
| `\|\|` | 逻辑 OR | MySQL 中 `\|\|` 是 OR！ |
| `GROUP_CONCAT(col SEPARATOR ',')` | `STRING_AGG(col, ',')` | 函数名不同 |
| `IFNULL(a, b)` | `COALESCE(a, b)` | PG 推荐标准 COALESCE |
| `IF(cond, a, b)` | `CASE WHEN cond THEN a ELSE b END` | PG 无 IF 函数 |
| `LENGTH(str)` | `LENGTH(str)` 或 `CHAR_LENGTH(str)` | PG LENGTH 也是字符数 |
| `SUBSTRING(str, pos, len)` | `SUBSTRING(str FROM pos FOR len)` | 标准语法 |

### 日期时间

| MySQL | PostgreSQL | 说明 |
|-------|-----------|------|
| `NOW()` | `NOW()` 或 `CURRENT_TIMESTAMP` | 相同 |
| `CURDATE()` | `CURRENT_DATE` | 关键字不同 |
| `DATE_ADD(d, INTERVAL 1 DAY)` | `d + INTERVAL '1 day'` | PG 直接用 + |
| `DATE_SUB(d, INTERVAL 1 MONTH)` | `d - INTERVAL '1 month'` | PG 直接用 - |
| `DATEDIFF(a, b)` | `a - b`（返回整数天） | PG 日期相减返回整数 |
| `DATE_FORMAT(d, '%Y-%m-%d')` | `TO_CHAR(d, 'YYYY-MM-DD')` | 格式字符串不同！ |
| `STR_TO_DATE(s, '%Y-%m-%d')` | `TO_DATE(s, 'YYYY-MM-DD')` | 格式字符串不同 |
| `UNIX_TIMESTAMP()` | `EXTRACT(EPOCH FROM NOW())` | PG 用 EXTRACT |
| `FROM_UNIXTIME(ts)` | `TO_TIMESTAMP(ts)` | 函数名不同 |

#### 格式字符串对照

| 含义 | MySQL | PostgreSQL |
|------|-------|-----------|
| 四位年 | `%Y` | `YYYY` |
| 两位月 | `%m` | `MM` |
| 两位日 | `%d` | `DD` |
| 24小时 | `%H` | `HH24` |
| 分钟 | `%i` | `MI` |
| 秒 | `%s` | `SS` |
| 星期名 | `%W` | `Day` |
| 月份名 | `%M` | `Month` |

### 条件和类型转换

| MySQL | PostgreSQL | 说明 |
|-------|-----------|------|
| `CAST(x AS SIGNED)` | `CAST(x AS INTEGER)` 或 `x::INTEGER` | PG 无 SIGNED/UNSIGNED |
| `CAST(x AS CHAR)` | `CAST(x AS TEXT)` 或 `x::TEXT` | 类型名不同 |
| `CONVERT(x, type)` | `CAST(x AS type)` | PG 无 CONVERT |
| 隐式转换: `'123' + 0` | 必须: `CAST('123' AS INTEGER)` 或 `'123'::INTEGER` | PG 不允许隐式转换！ |

### 其他

| MySQL | PostgreSQL | 说明 |
|-------|-----------|------|
| `SHOW TABLES` | `\dt` (psql) 或 `pg_tables` | 无 SHOW 语法 |
| `SHOW COLUMNS FROM t` | `\d t` (psql) 或 `information_schema` | 无 SHOW 语法 |
| `DESCRIBE t` | `\d t` | 无 DESCRIBE |
| `SHOW CREATE TABLE t` | `pg_dump -t t --schema-only` | 无直接等价 |
| `AUTO_INCREMENT` 值查询 | `SELECT currval('seq')` | 不同机制 |
| `@@变量` | `current_setting('var')` 或 `SHOW var` | 不同语法 |
| 反引号 \`col\` | 双引号 "col" | 引用标识符 |
| `LIMIT 5, 10` | `LIMIT 10 OFFSET 5` | 参数顺序相反！ |
| `REGEXP` | `~` 或 `~*`(CI) | 运算符不同 |
| `RLIKE` | `~` | MySQL 的 RLIKE 别名 |
| `\n` in strings | `E'\n'` 或 `CHR(10)` | PG 默认不解释转义 |

## 高危迁移点（最容易出 bug 的地方）

### 1. 隐式类型转换消失

```sql
-- MySQL: 正常运行
SELECT * FROM users WHERE id = '123';       -- '123' 自动转为整数
SELECT * FROM users WHERE age > '18';       -- '18' 自动转为整数
SELECT 1 + '2';                             -- 结果: 3

-- PostgreSQL: 需要改写
SELECT * FROM users WHERE id = 123;         -- 显式整数
SELECT * FROM users WHERE age > 18;         -- 显式整数
SELECT 1 + CAST('2' AS INTEGER);            -- 或 1 + '2'::INTEGER
```

### 2. GROUP BY 严格度

```sql
-- MySQL 5.7-(ONLY_FULL_GROUP_BY=OFF): 合法
SELECT name, city, COUNT(*) FROM users GROUP BY city;

-- PostgreSQL: 报错！name 没有在 GROUP BY 中也不是聚合
-- 修复:
SELECT name, city, COUNT(*) FROM users GROUP BY name, city;
-- 或:
SELECT MIN(name), city, COUNT(*) FROM users GROUP BY city;
```

### 3. LIMIT 参数顺序

```sql
-- MySQL: LIMIT offset, count
SELECT * FROM users LIMIT 20, 10;  -- 跳过 20 条，取 10 条

-- PostgreSQL: LIMIT count OFFSET offset（顺序相反！）
SELECT * FROM users LIMIT 10 OFFSET 20;
```

### 4. 日期格式字符串

```sql
-- MySQL
SELECT DATE_FORMAT(NOW(), '%Y-%m-%d %H:%i:%s');

-- PostgreSQL（格式完全不同！）
SELECT TO_CHAR(NOW(), 'YYYY-MM-DD HH24:MI:SS');
```

### 5. 布尔值

```sql
-- MySQL: 0/1 就是布尔
SELECT * FROM users WHERE active = 1;
INSERT INTO users (active) VALUES (1);

-- PostgreSQL: 真正的布尔
SELECT * FROM users WHERE active = TRUE;   -- 或 WHERE active
INSERT INTO users (active) VALUES (TRUE);
-- 但 PG 也接受 0/1 的隐式转换到布尔
```

> **重要**: PostgreSQL 在 WHERE 条件中允许 `WHERE active = 1`（通过隐式转换），但在 INSERT VALUES 上下文中**不会**隐式将整数 0/1 转换为 BOOLEAN。即 `INSERT INTO t (active) VALUES (1)` 会报类型错误，必须使用 `VALUES (TRUE)` 或 `VALUES (1::BOOLEAN)`。这是迁移中最常见的 BOOLEAN 相关陷阱。

## PostgreSQL 版本特性提示

| 版本 | 特性 | 迁移相关说明 |
|------|------|-------------|
| PostgreSQL 15 | 标准 MERGE 语句 | `MERGE INTO ... USING ... ON ... WHEN MATCHED THEN UPDATE WHEN NOT MATCHED THEN INSERT`，替代了之前的 UPSERT 变通方案（INSERT ... ON CONFLICT 仍推荐用于简单场景） |
| PostgreSQL 17 | `JSON_TABLE()` | 将 JSON 数组展开为关系表行，类似 MySQL 的 `JSON_TABLE`。此前 PG 用户需要 `jsonb_to_recordset` + LATERAL 等变通方式 |
