# SQL:1999 标准 (SQL3)

SQL:1999（ISO 9075:1999）是 SQL 标准的第三次重大修订，也是变化最大的一次。它将 SQL 从一个纯关系查询语言扩展为"对象-关系"语言，引入了递归查询、用户自定义类型、触发器标准化、正则匹配等重要特性。同时废弃了 SQL-92 的三级合规模型，改为"核心 + 可选特性包"模式。

## 合规模型变化

SQL-92 的 Entry/Intermediate/Full 三级模型被替换为：

- **Core SQL**（核心）：所有引擎必须实现的最小集
- **可选特性包**：编号为 T、F、S 等，例如 T121（WITH RECURSIVE）、T331（BASIC ROLES）

此模型一直沿用到 SQL:2023。

## BOOLEAN 类型

```sql
-- SQL:1999 标准
CREATE TABLE users (
    id      INTEGER PRIMARY KEY,
    active  BOOLEAN DEFAULT TRUE
);

SELECT * FROM users WHERE active = TRUE;
SELECT * FROM users WHERE active IS TRUE;   -- 区分 TRUE / FALSE / UNKNOWN
```

| 引擎 | BOOLEAN 支持 | 实现方式 |
|------|-------------|---------|
| MySQL | 部分 | `BOOLEAN` 是 `TINYINT(1)` 的别名，0=FALSE/1=TRUE，但允许其他整数值 |
| PostgreSQL | 完整 | 原生 BOOLEAN 类型，严格三值（TRUE/FALSE/NULL） |
| Oracle | 很晚 | 23c 才支持表列 BOOLEAN，之前只有 PL/SQL 中可用 |
| SQL Server | 部分 | 无 BOOLEAN，使用 `BIT`（0/1，无 UNKNOWN） |
| SQLite | 部分 | 无原生类型，0/1 作为布尔值 |
| BigQuery | 完整 | 原生 BOOL 类型 |
| Snowflake | 完整 | 原生 BOOLEAN 类型 |
| ClickHouse | 部分 | `Bool` 类型（UInt8 别名） |

> 这是一个典型案例：标准在 1999 年定义了特性，但 Oracle 直到 2023 年才完整支持，MySQL 至今仍是伪布尔。引擎开发者建议实现真正的三值 BOOLEAN。

## LOB 类型 (BLOB, CLOB)

```sql
CREATE TABLE documents (
    id      INTEGER PRIMARY KEY,
    content CLOB,           -- 字符大对象
    data    BLOB            -- 二进制大对象
);
```

| 引擎 | CLOB | BLOB |
|------|------|------|
| MySQL | LONGTEXT (最大 4GB) | LONGBLOB (最大 4GB) |
| PostgreSQL | TEXT (无限) | BYTEA (最大 1GB) 或 Large Object |
| Oracle | CLOB (最大 128TB) | BLOB (最大 128TB) |
| SQL Server | VARCHAR(MAX) / NVARCHAR(MAX) | VARBINARY(MAX) |
| SQLite | TEXT | BLOB |

## ARRAY 类型

```sql
-- SQL:1999 标准
CREATE TABLE survey (
    id      INTEGER PRIMARY KEY,
    tags    VARCHAR(50) ARRAY[10]
);

INSERT INTO survey VALUES (1, ARRAY['sql', 'database', 'standard']);
SELECT tags[1] FROM survey;  -- 标准数组下标从 1 开始
```

| 引擎 | ARRAY 支持 | 语法 |
|------|-----------|------|
| PostgreSQL | 完整 | `text[]`、`integer[]`，支持多维数组 |
| Oracle | 部分 | 需通过 VARRAY 或嵌套表实现 |
| BigQuery | 完整 | `ARRAY<STRING>`，标准函数 |
| Snowflake | 完整 | `ARRAY`（半结构化，元素为 VARIANT） |
| ClickHouse | 完整 | `Array(String)`，丰富的数组函数 |
| MySQL | ✗ | 不支持（可用 JSON 数组替代） |
| SQL Server | ✗ | 不支持 |
| SQLite | ✗ | 不支持 |

## 用户自定义类型 (UDT)

```sql
-- SQL:1999 标准: Distinct Type
CREATE TYPE currency AS DECIMAL(15,2) FINAL;

-- SQL:1999 标准: Structured Type
CREATE TYPE address AS (
    street  VARCHAR(100),
    city    VARCHAR(50),
    zip     CHAR(10)
) NOT FINAL;
```

实际支持情况很有限：
- **PostgreSQL**：最接近标准，支持 `CREATE TYPE` 创建复合类型、枚举类型、范围类型
- **Oracle**：通过 `CREATE TYPE ... AS OBJECT` 支持对象类型
- **其他引擎**：基本不支持或仅支持枚举类型（MySQL 的 `ENUM`）

> UDT 是 SQL:1999 中最"学术"的特性之一，实际采纳率很低。引擎开发者通常不需要优先实现。

## 递归查询 (WITH RECURSIVE)

这是 SQL:1999 中对实际应用影响最大的特性之一：

```sql
-- 标准语法: 查询组织架构树
WITH RECURSIVE org_tree AS (
    -- 锚点成员 (非递归部分)
    SELECT id, name, manager_id, 1 AS level
    FROM employees
    WHERE manager_id IS NULL

    UNION ALL

    -- 递归成员
    SELECT e.id, e.name, e.manager_id, t.level + 1
    FROM employees e
    INNER JOIN org_tree t ON e.manager_id = t.id
)
SELECT * FROM org_tree;
```

**WITH RECURSIVE 支持矩阵：**

| 引擎 | 支持 | 首次支持版本 | 关键字 |
|------|------|------------|-------|
| PostgreSQL | 完整 | 8.4 (2009) | `WITH RECURSIVE` |
| SQL Server | 完整 | 2005 | `WITH`（不需要 RECURSIVE 关键字） |
| Oracle | 部分 | 11g R2 (2009) | `WITH RECURSIVE`（也支持传统的 `CONNECT BY`） |
| MySQL | 完整 | 8.0 (2018) | `WITH RECURSIVE` |
| SQLite | 完整 | 3.8.3 (2014) | `WITH RECURSIVE` |
| BigQuery | 完整 | 2020 | `WITH RECURSIVE` |
| Snowflake | 完整 | 2019 | `WITH RECURSIVE` |
| ClickHouse | 实验性 | 22.4 (2022) | 需启用 `allow_experimental_analyzer` |

> **Oracle 的 CONNECT BY** 是 SQL:1999 之前的私有递归语法，至今仍广泛使用。迁移到标准 CTE 时需注意 `LEVEL`、`SYS_CONNECT_BY_PATH`、`CONNECT_BY_ROOT` 等 Oracle 特有伪列/函数。

## 触发器标准化

SQL:1999 首次标准化了触发器语法（之前各引擎各自实现）：

```sql
-- SQL:1999 标准
CREATE TRIGGER audit_update
    AFTER UPDATE ON employees
    REFERENCING OLD AS old_row NEW AS new_row
    FOR EACH ROW
    BEGIN ATOMIC
        INSERT INTO audit_log (emp_id, old_salary, new_salary, changed_at)
        VALUES (old_row.id, old_row.salary, new_row.salary, CURRENT_TIMESTAMP);
    END;
```

各引擎语法差异极大：

| 引擎 | 引用旧/新行 | 语句块 | 多事件 |
|------|-----------|-------|-------|
| MySQL | `OLD.col` / `NEW.col` | `BEGIN ... END` | 每个事件一个触发器 |
| PostgreSQL | `OLD.col` / `NEW.col` | 触发器函数 `CREATE FUNCTION ... RETURNS TRIGGER` | 多事件 |
| Oracle | `:OLD.col` / `:NEW.col` | PL/SQL 块 | 多事件 (`BEFORE INSERT OR UPDATE`) |
| SQL Server | `INSERTED` / `DELETED` 伪表 | T-SQL 块 | 多事件 |
| SQLite | `OLD.col` / `NEW.col` | 单条语句或 `BEGIN ... END` | 每个事件一个 |

## ROLE（角色）

```sql
-- SQL:1999 标准
CREATE ROLE analyst;
GRANT SELECT ON schema1.* TO analyst;
GRANT analyst TO user1;
```

所有主流引擎均支持角色，但语法和行为有差异。MySQL 在 8.0 才引入角色，且默认角色需要显式激活（`SET DEFAULT ROLE`）。

## SIMILAR TO（正则匹配）

```sql
-- SQL:1999 标准
SELECT * FROM t WHERE name SIMILAR TO '%(abc|def)%';
```

| 引擎 | SIMILAR TO | 替代语法 |
|------|-----------|---------|
| PostgreSQL | ✓ | `~` 操作符（POSIX 正则） |
| Oracle | ✗ | `REGEXP_LIKE()` |
| MySQL | ✗ | `REGEXP` / `RLIKE` |
| SQL Server | ✗ | `LIKE` + 有限模式（`[a-z]`） |
| BigQuery | ✗ | `REGEXP_CONTAINS()` |

> SIMILAR TO 的采纳率很低。PostgreSQL 是主要实现者，但即便 PostgreSQL 社区也更推荐使用 POSIX 正则（`~` 操作符）。标准本身在后续版本中也弱化了 SIMILAR TO。

## ROW 类型

```sql
-- SQL:1999 标准
SELECT ROW(1, 'Alice', 30) = ROW(1, 'Alice', 30);  -- TRUE

-- 行值构造器在 WHERE 中的用法
SELECT * FROM t WHERE (a, b) = (1, 2);
-- 等价于 WHERE a = 1 AND b = 2
```

行值比较在 PostgreSQL、MySQL（有限）、SQLite 中可用，常用于分页优化：

```sql
-- 基于游标的分页（比 OFFSET 更高效）
SELECT * FROM t WHERE (created_at, id) > ('2024-01-01', 100) ORDER BY created_at, id LIMIT 20;
```

## LATERAL

```sql
-- SQL:1999 标准: LATERAL 子查询可以引用前面 FROM 项的列
SELECT d.name, top_emp.name, top_emp.salary
FROM departments d,
LATERAL (
    SELECT e.name, e.salary
    FROM employees e
    WHERE e.dept_id = d.id
    ORDER BY e.salary DESC
    FETCH FIRST 1 ROW ONLY
) AS top_emp;
```

| 引擎 | LATERAL 支持 | 首次版本 |
|------|-------------|---------|
| PostgreSQL | ✓ | 9.3 (2013) |
| MySQL | ✓ | 8.0.14 (2019) |
| Oracle | ✓ | 12c (2013) |
| SQL Server | ✓ | 通过 `CROSS APPLY` / `OUTER APPLY` |
| BigQuery | ✓ | 原生支持 |
| Snowflake | ✓ | `LATERAL` + `FLATTEN` |

> SQL Server 的 `CROSS APPLY` / `OUTER APPLY` 功能上等同于 `LATERAL`，但语法不同。

## 对引擎开发者的实现建议

1. **WITH RECURSIVE 是最高优先级**。递归 CTE 在树/图遍历、层次查询中不可替代，几乎所有现代应用都会用到。
2. **非递归 CTE（WITH 子句）也是必须的**。它是代码组织的基础工具，所有 BI 工具和 ORM 都会生成。
3. **BOOLEAN 类型建议从一开始就实现为真三值类型**。避免 MySQL 的 TINYINT(1) 陷阱。
4. **触发器可以延后**。分布式引擎通常不支持触发器，分析引擎也不需要。如果要实现，参考 PostgreSQL 的触发器函数模型。
5. **ARRAY 类型推荐实现**。特别是分析型引擎，数组在数据处理中非常常用。
6. **UDT 和 SIMILAR TO 优先级低**。实际采纳率不高，可以暂缓。
7. **LATERAL 对复杂查询很有价值**。建议在实现了基本 JOIN 和子查询后尽早支持。
