# SQL:2003 标准

SQL:2003（ISO 9075:2003）是一个里程碑式版本，引入了现代 SQL 中最常用的特性之一——窗口函数。同时还引入了 MERGE INTO、IDENTITY 列、序列、XML 类型等重要功能。这个版本标志着 SQL 从"查询语言"向"数据分析语言"的转型。

## 窗口函数 (Window Functions)

窗口函数是 SQL:2003 中影响最深远的特性，它允许在不折叠行的情况下执行聚合计算：

```sql
-- SQL:2003 标准语法
SELECT
    employee_id,
    department_id,
    salary,
    -- 聚合窗口函数
    SUM(salary) OVER (PARTITION BY department_id) AS dept_total,
    AVG(salary) OVER (PARTITION BY department_id) AS dept_avg,
    -- 排名函数
    ROW_NUMBER() OVER (PARTITION BY department_id ORDER BY salary DESC) AS rank_in_dept,
    RANK() OVER (ORDER BY salary DESC) AS overall_rank,
    DENSE_RANK() OVER (ORDER BY salary DESC) AS dense_overall_rank,
    -- 偏移函数
    LAG(salary, 1) OVER (ORDER BY hire_date) AS prev_salary,
    LEAD(salary, 1) OVER (ORDER BY hire_date) AS next_salary,
    FIRST_VALUE(salary) OVER (PARTITION BY department_id ORDER BY salary DESC) AS max_salary,
    LAST_VALUE(salary) OVER (
        PARTITION BY department_id ORDER BY salary
        ROWS BETWEEN UNBOUNDED PRECEDING AND UNBOUNDED FOLLOWING
    ) AS max_salary_alt,
    -- 分布函数
    NTILE(4) OVER (ORDER BY salary) AS quartile,
    PERCENT_RANK() OVER (ORDER BY salary) AS pct_rank,
    CUME_DIST() OVER (ORDER BY salary) AS cum_dist
FROM employees;
```

### 窗口帧 (Window Frame)

```sql
-- ROWS 帧: 基于物理行偏移
SUM(amount) OVER (ORDER BY order_date ROWS BETWEEN 2 PRECEDING AND CURRENT ROW)

-- RANGE 帧: 基于逻辑值范围
SUM(amount) OVER (ORDER BY order_date RANGE BETWEEN INTERVAL '7' DAY PRECEDING AND CURRENT ROW)

-- GROUPS 帧 (SQL:2011 新增): 基于 peer 组
SUM(amount) OVER (ORDER BY order_date GROUPS BETWEEN 1 PRECEDING AND 1 FOLLOWING)
```

**默认帧行为（一个极重要的细节）：**

当指定 `ORDER BY` 时，默认帧是 `RANGE BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW`，而不是很多人以为的 `ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW`。这在有重复值时会导致不同的结果。

```sql
-- 命名窗口 (WINDOW 子句)，减少重复定义
SELECT
    ROW_NUMBER() OVER w AS rn,
    SUM(salary) OVER w AS running_total
FROM employees
WINDOW w AS (PARTITION BY dept_id ORDER BY hire_date);
```

## MERGE INTO

MERGE 语句（也称 UPSERT 的标准形式）允许在一条语句中组合 INSERT、UPDATE、DELETE：

```sql
-- SQL:2003 标准
MERGE INTO target_table t
USING source_table s
ON t.id = s.id
WHEN MATCHED THEN
    UPDATE SET t.name = s.name, t.value = s.value
WHEN NOT MATCHED THEN
    INSERT (id, name, value) VALUES (s.id, s.name, s.value);
```

**MERGE 支持矩阵：**

| 引擎 | 标准 MERGE | 替代语法 |
|------|-----------|---------|
| Oracle | ✓ (9i+) | — |
| SQL Server | ✓ (2008+) | — |
| PostgreSQL | ✓ (15+, 2022) | `INSERT ... ON CONFLICT` (9.5+) |
| MySQL | ✗ | `INSERT ... ON DUPLICATE KEY UPDATE`、`REPLACE INTO` |
| SQLite | ✗ | `INSERT ... ON CONFLICT` (3.24+) |
| BigQuery | ✓ | — |
| Snowflake | ✓ | — |
| ClickHouse | ✗ | ReplacingMergeTree 引擎 |

> **MySQL 始终没有实现标准 MERGE**，而是使用 `INSERT ... ON DUPLICATE KEY UPDATE`。这是 MySQL 与标准的一个显著分歧。PostgreSQL 在 2022 年才正式支持 MERGE（非常晚），之前一直推荐使用 `ON CONFLICT` 子句。

## IDENTITY 列

```sql
-- SQL:2003 标准
CREATE TABLE orders (
    id    INTEGER GENERATED ALWAYS AS IDENTITY,
    name  VARCHAR(100)
);

-- 变体
CREATE TABLE orders2 (
    id    INTEGER GENERATED BY DEFAULT AS IDENTITY (START WITH 1 INCREMENT BY 1),
    name  VARCHAR(100)
);
```

| 引擎 | 标准 IDENTITY | 非标准替代 |
|------|-------------|----------|
| PostgreSQL | ✓ (10+) | `SERIAL` (旧语法，基于序列) |
| Oracle | ✓ (12c+) | `CREATE SEQUENCE` + 触发器（旧方式） |
| SQL Server | 非标准 | `IDENTITY(1,1)` 属性（语法不同但概念相同） |
| MySQL | ✗ | `AUTO_INCREMENT` |
| SQLite | ✗ | `INTEGER PRIMARY KEY`（隐式自增）、`AUTOINCREMENT` |
| BigQuery | ✗ | 无自增列概念 |
| Snowflake | ✓ | `AUTOINCREMENT` |

> `GENERATED ALWAYS` 表示始终由系统生成，不允许手动指定值。`GENERATED BY DEFAULT` 表示默认由系统生成，但允许显式插入值。这个区别在数据迁移时很重要。

## CREATE SEQUENCE

```sql
-- SQL:2003 标准
CREATE SEQUENCE order_seq
    START WITH 1
    INCREMENT BY 1
    NO MINVALUE
    NO MAXVALUE
    NO CYCLE;

-- 使用序列
SELECT NEXT VALUE FOR order_seq;        -- 标准语法
INSERT INTO orders (id, name) VALUES (NEXT VALUE FOR order_seq, 'test');
```

| 引擎 | 支持 | 获取下一个值语法 |
|------|------|---------------|
| PostgreSQL | ✓ | `nextval('order_seq')` |
| Oracle | ✓ | `order_seq.NEXTVAL` |
| SQL Server | ✓ (2012+) | `NEXT VALUE FOR order_seq` |
| MySQL | ✗ | 不支持（依赖 AUTO_INCREMENT） |
| SQLite | ✗ | 不支持 |
| BigQuery | ✗ | 不支持 |
| Snowflake | ✓ | `order_seq.NEXTVAL` |

## XML 类型和函数

```sql
-- SQL:2003 标准
CREATE TABLE configs (
    id   INTEGER,
    data XML
);

-- 标准 XML 函数
SELECT XMLELEMENT(NAME "emp", XMLATTRIBUTES(name AS "name"), salary) FROM employees;
SELECT XMLQUERY('/employees/name' PASSING data) FROM configs;
```

XML 支持在所有引擎中都比较碎片化。PostgreSQL、Oracle、SQL Server 有较完整的 XML 支持，但各自的函数集不同。随着 JSON 的兴起，XML 在新引擎（BigQuery、ClickHouse、Snowflake）中基本不被支持。

## BIGINT

```sql
CREATE TABLE large_ids (
    id BIGINT  -- 8字节整数，范围 -2^63 到 2^63-1
);
```

SQL:2003 正式将 BIGINT 加入标准。所有主流引擎均支持 BIGINT 或等价类型。Oracle 使用 `NUMBER(19)` 或 21c+ 原生 BIGINT。

## MULTISET

```sql
-- SQL:2003 标准
CREATE TABLE t (
    id   INTEGER,
    tags VARCHAR(50) MULTISET  -- 允许重复的集合
);
```

MULTISET 是 SQL:2003 中最少被实现的特性之一。Oracle 通过嵌套表有部分支持，其他引擎基本不支持。对于引擎开发者来说，这个特性优先级极低。

## FILTER 子句

```sql
-- SQL:2003 标准: 条件聚合的标准语法
SELECT
    COUNT(*) AS total,
    COUNT(*) FILTER (WHERE status = 'active') AS active_count,
    SUM(amount) FILTER (WHERE category = 'A') AS category_a_total
FROM orders;
```

| 引擎 | FILTER 支持 | 替代方案 |
|------|------------|---------|
| PostgreSQL | ✓ (9.4+) | — |
| SQLite | ✓ (3.30+) | — |
| MySQL | ✗ | `SUM(CASE WHEN status = 'active' THEN 1 ELSE 0 END)` |
| Oracle | ✗ | 同上 CASE 方式 |
| SQL Server | ✗ | 同上 CASE 方式 |
| BigQuery | ✓ | `COUNTIF()` 等专用函数 |
| Snowflake | ✗ | CASE 方式 |
| ClickHouse | ✗ | `countIf()`、`sumIf()` 等 `-If` 后缀函数 |

> FILTER 子句语义清晰，远优于 CASE WHEN 的变通写法。但采纳率不高。PostgreSQL 和 SQLite 是标准的忠实实现者。

## 标准语法 vs 各引擎实际语法

### 自增 ID 的实现差异

这是各引擎差异最大的领域之一：

```sql
-- SQL:2003 标准
CREATE TABLE t (id INTEGER GENERATED ALWAYS AS IDENTITY PRIMARY KEY);

-- PostgreSQL (旧)
CREATE TABLE t (id SERIAL PRIMARY KEY);

-- PostgreSQL (新，推荐)
CREATE TABLE t (id INTEGER GENERATED ALWAYS AS IDENTITY PRIMARY KEY);

-- MySQL
CREATE TABLE t (id INT AUTO_INCREMENT PRIMARY KEY);

-- SQL Server
CREATE TABLE t (id INT IDENTITY(1,1) PRIMARY KEY);

-- Oracle (旧)
CREATE TABLE t (id NUMBER PRIMARY KEY);
CREATE SEQUENCE t_seq;
-- 然后用触发器填充

-- Oracle (新，12c+)
CREATE TABLE t (id NUMBER GENERATED ALWAYS AS IDENTITY PRIMARY KEY);

-- SQLite
CREATE TABLE t (id INTEGER PRIMARY KEY);  -- 隐式 ROWID 自增
```

## 对引擎开发者的实现建议

1. **窗口函数是最高优先级**。它是现代 SQL 的核心特性，BI 工具、数据分析查询、分页优化都离不开它。建议分步实现：先实现 ROW_NUMBER/RANK/DENSE_RANK → 然后 LAG/LEAD/FIRST_VALUE/LAST_VALUE → 最后完整帧支持（ROWS/RANGE）。
2. **注意默认帧行为**。`ORDER BY` 存在时的默认帧是 `RANGE BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW`，这是一个常见的实现错误。
3. **MERGE 建议实现**。它是标准的 upsert 方案。如果你的引擎兼容 MySQL，优先实现 `ON DUPLICATE KEY UPDATE`；如果兼容 PostgreSQL，优先实现 `ON CONFLICT`。
4. **SEQUENCE 比 IDENTITY 更灵活**。如果要实现自增 ID，建议底层用序列实现，IDENTITY 作为语法糖。
5. **FILTER 子句实现成本低**。它本质上是条件聚合的语法糖，可以在优化器中转换为 CASE WHEN，但对用户体验提升很大。
6. **XML 可以跳过**。除非你的引擎面向企业市场（银行、保险等仍在使用大量 XML），否则把精力投入 JSON 支持更有价值。
7. **BIGINT 是必须的**。8 字节整数在现代系统中是基本需求（ID 生成、时间戳存储等）。
