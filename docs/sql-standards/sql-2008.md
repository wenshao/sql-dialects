# SQL:2008 标准

SQL:2008（ISO 9075:2008）引入了几个非常实用的特性，其中标准分页语法（`FETCH FIRST N ROWS ONLY`）和 `TRUNCATE TABLE` 解决了长期以来各引擎各自为政的问题。这个版本虽然不像 SQL:2003 那样有窗口函数这样的"重磅炸弹"，但引入的特性都是日常开发中高频使用的。

## TRUNCATE TABLE

```sql
-- SQL:2008 标准
TRUNCATE TABLE orders;

-- 带选项
TRUNCATE TABLE orders CONTINUE IDENTITY;    -- 不重置自增计数器
TRUNCATE TABLE orders RESTART IDENTITY;     -- 重置自增计数器
TRUNCATE TABLE orders CASCADE;              -- 级联截断引用表
```

在 SQL:2008 之前，TRUNCATE 已经被各引擎广泛实现（Oracle 从 7.x 就支持了），但标准直到 2008 年才收录。

| 引擎 | TRUNCATE 支持 | RESTART/CONTINUE IDENTITY | CASCADE |
|------|-------------|--------------------------|---------|
| MySQL | ✓ | 隐式重置自增 | ✗ (需先删除外键) |
| PostgreSQL | ✓ | ✓ | ✓ |
| Oracle | ✓ | ✗ | ✓ (12c+) |
| SQL Server | ✓ | 隐式重置 IDENTITY | ✗ |
| SQLite | ✗ (用 DELETE) | — | — |
| BigQuery | ✓ | — | — |
| Snowflake | ✓ | — | — |
| ClickHouse | ✓ | — | — |

> SQLite 没有 TRUNCATE 命令，使用 `DELETE FROM table` 代替。当没有触发器时，SQLite 会自动优化为类似 TRUNCATE 的操作。

TRUNCATE 与 DELETE 的关键区别：
- TRUNCATE 不触发行级触发器（DELETE 触发）
- TRUNCATE 不可回滚（大多数引擎中是 DDL 操作）—— 但 PostgreSQL 的 TRUNCATE 是事务性的，可以回滚
- TRUNCATE 通常比 DELETE 快得多（不逐行删除，直接释放数据页）
- TRUNCATE 一般会重置自增计数器

## FETCH FIRST N ROWS ONLY（标准分页）

这是 SQL 历史上最"迟到"的标准化之一。分页是每个应用都需要的功能，但直到 2008 年才有标准语法：

```sql
-- SQL:2008 标准语法
SELECT * FROM products
ORDER BY price DESC
OFFSET 20 ROWS
FETCH FIRST 10 ROWS ONLY;

-- 变体: FETCH NEXT 等价于 FETCH FIRST
SELECT * FROM products
ORDER BY price DESC
OFFSET 20 ROWS
FETCH NEXT 10 ROWS ONLY;

-- 只取前 N 行（无偏移）
SELECT * FROM products
ORDER BY price DESC
FETCH FIRST 5 ROWS ONLY;

-- 带 TIES（包含同值行）
SELECT * FROM products
ORDER BY price DESC
FETCH FIRST 5 ROWS WITH TIES;
```

### 各引擎分页语法对比

这是 SQL 中语法碎片化最严重的领域：

```sql
-- SQL:2008 标准
SELECT * FROM t ORDER BY id OFFSET 20 ROWS FETCH FIRST 10 ROWS ONLY;

-- MySQL / MariaDB / SQLite / ClickHouse
SELECT * FROM t ORDER BY id LIMIT 10 OFFSET 20;

-- PostgreSQL (两种都支持)
SELECT * FROM t ORDER BY id LIMIT 10 OFFSET 20;         -- 非标准但更常用
SELECT * FROM t ORDER BY id OFFSET 20 FETCH FIRST 10 ROWS ONLY;  -- 标准

-- Oracle (传统, 12c 之前)
SELECT * FROM (
    SELECT t.*, ROWNUM rn FROM (
        SELECT * FROM t ORDER BY id
    ) t WHERE ROWNUM <= 30
) WHERE rn > 20;

-- Oracle (12c+, 支持标准语法)
SELECT * FROM t ORDER BY id OFFSET 20 ROWS FETCH FIRST 10 ROWS ONLY;

-- SQL Server (传统, 2012 之前)
SELECT TOP 10 * FROM t WHERE id NOT IN (SELECT TOP 20 id FROM t ORDER BY id) ORDER BY id;

-- SQL Server (2012+, 支持标准语法)
SELECT * FROM t ORDER BY id OFFSET 20 ROWS FETCH NEXT 10 ROWS ONLY;

-- BigQuery
SELECT * FROM t ORDER BY id LIMIT 10 OFFSET 20;

-- Snowflake (两种都支持)
SELECT * FROM t ORDER BY id LIMIT 10 OFFSET 20;
SELECT * FROM t ORDER BY id OFFSET 20 ROWS FETCH FIRST 10 ROWS ONLY;
```

**分页语法支持矩阵：**

| 语法 | MySQL | PostgreSQL | Oracle | SQL Server | SQLite | BigQuery | Snowflake | ClickHouse |
|------|-------|-----------|--------|-----------|--------|----------|-----------|------------|
| FETCH FIRST N ROWS ONLY | ✗ | ✓ (8.4+) | ✓ (12c+) | ✓ (2012+) | ✗ | ✗ | ✓ | ✗ |
| LIMIT ... OFFSET | ✓ | ✓ | ✗ | ✗ | ✓ | ✓ | ✓ | ✓ |
| TOP N | ✗ | ✗ | ✗ | ✓ | ✗ | ✗ | ✓ | ✗ |
| ROWNUM | ✗ | ✗ | ✓ | ✗ | ✗ | ✗ | ✗ | ✗ |
| WITH TIES | ✗ | ✓ (13+) | ✓ (12c+) | ✓ (2012+) | ✗ | ✗ | ✓ | ✗ |
| PERCENT | ✗ | ✗ | ✓ (12c+) | ✓ | ✗ | ✗ | ✗ | ✗ |

> **MySQL 至今不支持标准分页语法**，只支持 `LIMIT ... OFFSET`。这是 MySQL 与标准最显著的差异之一。

### OFFSET 分页的性能问题

无论使用哪种语法，大偏移量的分页（`OFFSET 100000`）在所有引擎中都有性能问题——需要先跳过大量行。更好的方案是基于游标（keyset）分页：

```sql
-- 不推荐: OFFSET 分页（大偏移量时慢）
SELECT * FROM orders ORDER BY id LIMIT 20 OFFSET 100000;

-- 推荐: 游标分页（始终快速）
SELECT * FROM orders WHERE id > 100000 ORDER BY id LIMIT 20;

-- 多列排序的游标分页
SELECT * FROM orders
WHERE (created_at, id) > ('2024-01-15', 5000)
ORDER BY created_at, id
LIMIT 20;
```

## INSTEAD OF 触发器

```sql
-- SQL:2008 标准化了 INSTEAD OF 触发器（用于可更新视图）
CREATE VIEW active_employees AS
    SELECT * FROM employees WHERE status = 'active';

CREATE TRIGGER instead_insert_active_emp
    INSTEAD OF INSERT ON active_employees
    FOR EACH ROW
    BEGIN
        INSERT INTO employees (name, status) VALUES (NEW.name, 'active');
    END;
```

| 引擎 | INSTEAD OF 触发器 | 说明 |
|------|------------------|------|
| SQL Server | ✓ | 最早实现（SQL Server 2000） |
| PostgreSQL | ✓ | 支持，也可用 RULE 系统替代 |
| Oracle | ✓ | 支持 |
| SQLite | ✓ | 支持 |
| MySQL | ✗ | 不支持 |
| BigQuery | ✗ | 不支持 |
| Snowflake | ✗ | 不支持 |

## MERGE 增强

SQL:2008 在 SQL:2003 的 MERGE 基础上增加了 DELETE 子句：

```sql
-- SQL:2008 增强: MERGE 中的 DELETE
MERGE INTO inventory t
USING daily_shipments s
ON t.product_id = s.product_id
WHEN MATCHED AND t.quantity + s.quantity <= 0 THEN
    DELETE                                          -- SQL:2008 新增
WHEN MATCHED THEN
    UPDATE SET t.quantity = t.quantity + s.quantity
WHEN NOT MATCHED THEN
    INSERT (product_id, quantity) VALUES (s.product_id, s.quantity);
```

Oracle 和 SQL Server 支持 MERGE 中的 DELETE 子句。PostgreSQL 15+ 的 MERGE 也支持。

## 其他 SQL:2008 新增

- **ORDER BY 在子查询中**：标准正式允许子查询中使用 ORDER BY（之前是未定义行为）
- **CASE 表达式增强**：允许更复杂的 CASE 条件
- **诊断增强**：改进了错误处理和诊断信息的标准化

## 对引擎开发者的实现建议

1. **分页语法是最高优先级**。如果你的引擎面向应用开发者，分页是最常用的功能之一。建议同时支持 `LIMIT ... OFFSET`（用户习惯）和标准的 `FETCH FIRST N ROWS ONLY`（标准合规）。
2. **TRUNCATE 实现简单但很实用**。比 DELETE 快很多，测试和开发环境中高频使用。注意决定它是 DDL（不可回滚）还是 DML（可回滚，PostgreSQL 风格）。
3. **WITH TIES 是一个加分项**。它解决了"取 Top N 但不想丢弃同分记录"的常见需求，实现成本不高。
4. **INSTEAD OF 触发器**。如果你的引擎支持视图和触发器，INSTEAD OF 触发器可以让视图变得可更新，这对 ORM 框架很有价值。但如果触发器整体不在路线图中，这个可以跳过。
5. **为游标分页提供优化**。无论支持哪种分页语法，确保 `WHERE (a, b) > (v1, v2) ORDER BY a, b LIMIT n` 这种模式能使用索引，因为它是高性能分页的基础。
