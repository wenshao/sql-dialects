# SQL:2011 标准

SQL:2011（ISO 9075:2011）最重要的贡献是引入了时态表（Temporal Tables）标准。时态表允许数据库自动维护数据的历史版本，支持"某个时间点的数据是什么样的"这类查询。这个特性在审计、合规、数据回溯等场景中极为重要。

## 时态表概述

SQL:2011 定义了两种时态表：

| 类型 | 含义 | 维护方式 |
|------|------|---------|
| System-Versioned Temporal Table | 系统版本表，自动记录行的有效时间 | 系统自动维护（INSERT/UPDATE/DELETE 时自动记录） |
| Application-Time Period Table | 应用时间表，记录业务有效期 | 应用程序显式维护 |

两者可以组合使用（Bitemporal Table），同时追踪系统时间和业务时间。

## System-Versioned Temporal Table（系统版本表）

### 标准语法

```sql
-- SQL:2011 标准
CREATE TABLE employees (
    emp_id        INTEGER PRIMARY KEY,
    name          VARCHAR(100),
    department    VARCHAR(50),
    salary        DECIMAL(10,2),
    -- 系统时间列
    sys_start     TIMESTAMP(6) GENERATED ALWAYS AS ROW START,
    sys_end       TIMESTAMP(6) GENERATED ALWAYS AS ROW END,
    -- 声明系统时间周期
    PERIOD FOR SYSTEM_TIME (sys_start, sys_end)
) WITH SYSTEM VERSIONING;
```

当对表执行 DML 操作时：
- **INSERT**：设置 `sys_start = CURRENT_TIMESTAMP`，`sys_end = '9999-12-31'`
- **UPDATE**：旧行的 `sys_end` 设为当前时间并移入历史，新行的 `sys_start` 设为当前时间
- **DELETE**：行的 `sys_end` 设为当前时间并移入历史

### 时态查询语法

```sql
-- 查询某个时间点的数据快照
SELECT * FROM employees
FOR SYSTEM_TIME AS OF TIMESTAMP '2024-01-15 10:00:00';

-- 查询某个时间范围内存在过的所有版本
SELECT * FROM employees
FOR SYSTEM_TIME BETWEEN TIMESTAMP '2024-01-01' AND TIMESTAMP '2024-06-30';

-- 查询某个时间范围内的所有变更（不含端点）
SELECT * FROM employees
FOR SYSTEM_TIME FROM TIMESTAMP '2024-01-01' TO TIMESTAMP '2024-06-30';

-- 查询有交叠的行
SELECT * FROM employees
FOR SYSTEM_TIME CONTAINED IN (TIMESTAMP '2024-01-01', TIMESTAMP '2024-06-30');

-- 查看所有历史版本（包括当前行和已删除/修改的行）
SELECT * FROM employees FOR SYSTEM_TIME ALL;
```

### 各引擎实现对比

| 特性 | SQL Server | MariaDB | PostgreSQL | Oracle | MySQL | SQLite |
|------|-----------|---------|-----------|--------|-------|--------|
| System-Versioned Table | ✓ (2016+) | ✓ (10.3+) | 手动实现 | ✓ (12c+ Flashback) | ✗ | ✗ |
| FOR SYSTEM_TIME AS OF | ✓ | ✓ | ✗ | AS OF TIMESTAMP | ✗ | ✗ |
| FOR SYSTEM_TIME BETWEEN | ✓ | ✓ | ✗ | ✗ | ✗ | ✗ |
| 历史表分离 | ✓ (自动) | ✓ (分区) | — | ✓ (Undo 段) | — | — |
| 应用时间表 | ✓ (2016+) | ✓ (10.5+) | ✗ | ✗ | ✗ | ✗ |

#### SQL Server 的实现（最接近标准）

```sql
-- SQL Server 2016+ (最完善的标准实现)
CREATE TABLE employees (
    emp_id        INT PRIMARY KEY,
    name          NVARCHAR(100),
    department    NVARCHAR(50),
    salary        DECIMAL(10,2),
    sys_start     DATETIME2 GENERATED ALWAYS AS ROW START NOT NULL,
    sys_end       DATETIME2 GENERATED ALWAYS AS ROW END NOT NULL,
    PERIOD FOR SYSTEM_TIME (sys_start, sys_end)
) WITH (SYSTEM_VERSIONING = ON (HISTORY_TABLE = dbo.employees_history));

-- 查询历史
SELECT * FROM employees FOR SYSTEM_TIME AS OF '2024-01-15';

-- 关闭版本控制（便于维护）
ALTER TABLE employees SET (SYSTEM_VERSIONING = OFF);
```

#### MariaDB 的实现

```sql
-- MariaDB 10.3+
CREATE TABLE employees (
    emp_id        INT PRIMARY KEY,
    name          VARCHAR(100),
    department    VARCHAR(50),
    salary        DECIMAL(10,2)
) WITH SYSTEM VERSIONING;
-- MariaDB 自动添加隐藏的 row_start / row_end 列

-- 查询历史
SELECT * FROM employees FOR SYSTEM_TIME AS OF TIMESTAMP '2024-01-15 10:00:00';

-- 按分区管理历史数据（MariaDB 特有）
ALTER TABLE employees
    PARTITION BY SYSTEM_TIME INTERVAL 1 MONTH (
        PARTITION p_history HISTORY,
        PARTITION p_current CURRENT
    );
```

#### Oracle 的 Flashback（概念相似但语法不同）

```sql
-- Oracle Flashback Query (不是标准语法，但概念等价)
SELECT * FROM employees AS OF TIMESTAMP
    TO_TIMESTAMP('2024-01-15 10:00:00', 'YYYY-MM-DD HH24:MI:SS');

-- Oracle Flashback Version Query
SELECT * FROM employees
    VERSIONS BETWEEN TIMESTAMP
        TO_TIMESTAMP('2024-01-01', 'YYYY-MM-DD') AND
        TO_TIMESTAMP('2024-06-30', 'YYYY-MM-DD');
```

Oracle 的 Flashback 基于 Undo 段实现，功能强大但语法与标准不同，且受 Undo 保留时间限制。

#### PostgreSQL 的手动实现

PostgreSQL 没有原生的时态表支持，需要通过触发器手动实现：

```sql
-- PostgreSQL: 手动实现系统版本表
CREATE TABLE employees (
    emp_id      INTEGER PRIMARY KEY,
    name        VARCHAR(100),
    salary      DECIMAL(10,2)
);

CREATE TABLE employees_history (
    emp_id      INTEGER,
    name        VARCHAR(100),
    salary      DECIMAL(10,2),
    sys_start   TIMESTAMPTZ NOT NULL,
    sys_end     TIMESTAMPTZ NOT NULL DEFAULT 'infinity'
);

-- 需要自己写触发器来维护历史
-- 也可以使用社区扩展如 temporal_tables
```

## Application-Time Period Table（应用时间表）

应用时间表用于跟踪业务有效期（如保险合同的生效/失效日期）：

```sql
-- SQL:2011 标准
CREATE TABLE insurance_policies (
    policy_id     INTEGER,
    customer_id   INTEGER,
    coverage      VARCHAR(50),
    valid_from    DATE,
    valid_to      DATE,
    PERIOD FOR valid_period (valid_from, valid_to),
    PRIMARY KEY (policy_id, valid_period WITHOUT OVERLAPS)
);

-- 按时间段更新（标准自动处理时间段分裂）
UPDATE insurance_policies
FOR PORTION OF valid_period FROM DATE '2024-06-01' TO DATE '2024-12-31'
SET coverage = 'premium';

-- 按时间段删除
DELETE FROM insurance_policies
FOR PORTION OF valid_period FROM DATE '2024-06-01' TO DATE '2024-12-31';
```

应用时间表的支持更为有限：

| 特性 | SQL Server | MariaDB | 其他引擎 |
|------|-----------|---------|---------|
| PERIOD FOR (应用时间) | ✓ (2016+) | ✓ (10.5+) | ✗ |
| WITHOUT OVERLAPS | ✓ (2022+) | ✗ | PostgreSQL 17+ (约束排除) |
| FOR PORTION OF (UPDATE/DELETE) | ✗ | ✓ (10.5+) | ✗ |

> WITHOUT OVERLAPS 约束确保同一实体在同一时间段不会有重叠的记录，这对保险、合同等业务逻辑非常关键。手动实现这个约束需要复杂的触发器逻辑。

## Bitemporal Table（双时态表）

同时维护系统时间和应用时间：

```sql
-- SQL:2011 标准: 双时态表
CREATE TABLE contracts (
    contract_id   INTEGER,
    amount        DECIMAL(12,2),
    -- 业务有效期
    valid_from    DATE,
    valid_to      DATE,
    PERIOD FOR valid_period (valid_from, valid_to),
    -- 系统版本
    sys_start     TIMESTAMP(6) GENERATED ALWAYS AS ROW START,
    sys_end       TIMESTAMP(6) GENERATED ALWAYS AS ROW END,
    PERIOD FOR SYSTEM_TIME (sys_start, sys_end),
    PRIMARY KEY (contract_id, valid_period WITHOUT OVERLAPS)
) WITH SYSTEM VERSIONING;

-- 查询: 在 2024-03-15 这天，系统记录的 2024-01-01 时的合同状态
SELECT * FROM contracts
FOR SYSTEM_TIME AS OF TIMESTAMP '2024-03-15'
WHERE valid_from <= DATE '2024-01-01' AND valid_to > DATE '2024-01-01';
```

只有 SQL Server 2016+ 和 MariaDB 10.5+ 可以实现接近完整的双时态表。

## 其他 SQL:2011 新增

- **GROUPS 窗口帧**：`ROWS` 和 `RANGE` 之外的第三种帧类型，基于 peer 组
- **WINDOW 子句中的 EXCLUDE**：排除当前行或 peer 组
- **CALL 语句增强**：存储过程调用的标准化改进
- **LIKE_REGEX**：基于 XQuery 正则表达式的模式匹配（采纳率极低）

## 对引擎开发者的实现建议

1. **时态表是一个高价值但高复杂度的特性**。如果你的引擎面向企业应用（金融、保险、医疗、审计），时态表是差异化竞争力。否则可以延后。
2. **优先实现 System-Versioned 表**。它比 Application-Time 表更通用，实现也更直接（自动维护 start/end 时间戳 + 历史表）。
3. **参考 SQL Server 的实现**。它是目前最接近标准的实现，API 设计也最清晰。
4. **历史数据管理要提前设计**。历史表会持续增长，需要考虑分区（按时间）、保留策略（自动清理旧历史）、压缩等。MariaDB 的按 SYSTEM_TIME 分区是个好的参考。
5. **如果实现成本太高，可以先提供 AS OF 查询**。很多分析引擎（如 Snowflake、BigQuery）通过 Time Travel 功能提供了类似的回溯查询，不需要完整的时态表语义。
6. **FOR PORTION OF 语法实现复杂**。它需要自动分裂时间段，涉及一条 UPDATE 变成 INSERT + UPDATE + UPDATE 的场景，事务语义复杂。建议作为后期特性。
