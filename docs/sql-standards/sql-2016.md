# SQL:2016 标准

SQL:2016（ISO 9075:2016）是 JSON 时代最重要的 SQL 标准更新。它引入了标准化的 JSON 函数体系、LISTAGG 聚合函数、行模式识别（MATCH_RECOGNIZE）和多态表函数。其中 JSON 支持是这个版本最核心的贡献，但各引擎的实现差异也是最大的。

## JSON 支持

### 标准 JSON 函数体系

SQL:2016 定义了一套完整的 JSON 操作函数：

```sql
-- JSON_VALUE: 从 JSON 中提取标量值
SELECT JSON_VALUE('{"name": "Alice", "age": 30}', '$.name');
-- 返回: 'Alice'

-- JSON_QUERY: 从 JSON 中提取对象或数组
SELECT JSON_QUERY('{"addr": {"city": "NYC"}}', '$.addr');
-- 返回: '{"city": "NYC"}'

-- JSON_EXISTS: 检查路径是否存在
SELECT * FROM users WHERE JSON_EXISTS(profile, '$.email');

-- IS JSON: 检查值是否为合法 JSON
SELECT * FROM logs WHERE data IS JSON;

-- JSON_OBJECT: 构造 JSON 对象
SELECT JSON_OBJECT('name' VALUE name, 'age' VALUE age) FROM users;
-- 返回: '{"name": "Alice", "age": 30}'

-- JSON_ARRAY: 构造 JSON 数组
SELECT JSON_ARRAY(1, 2, 'three', NULL ABSENT ON NULL);
-- 返回: '[1, 2, "three"]'

-- JSON_OBJECTAGG: 聚合为 JSON 对象
SELECT JSON_OBJECTAGG(key VALUE val) FROM kv_pairs;

-- JSON_ARRAYAGG: 聚合为 JSON 数组
SELECT department, JSON_ARRAYAGG(name ORDER BY name) FROM employees GROUP BY department;
```

### JSON_TABLE: 将 JSON 转为关系表

JSON_TABLE 是最强大也最复杂的 JSON 函数，设计继承自 SQL:2006 的 XMLTABLE：

```sql
-- SQL:2016 标准
SELECT jt.*
FROM orders,
     JSON_TABLE(
         order_data,
         '$.items[*]'
         COLUMNS (
             item_id     INTEGER      PATH '$.id',
             item_name   VARCHAR(100) PATH '$.name',
             quantity     INTEGER      PATH '$.qty' DEFAULT 1 ON EMPTY,
             price       DECIMAL(10,2) PATH '$.price',
             -- 嵌套路径
             NESTED PATH '$.tags[*]' COLUMNS (
                 tag VARCHAR(50) PATH '$'
             )
         )
     ) AS jt;
```

### 各引擎 JSON 路径语法差异

```sql
-- SQL:2016 标准路径语法 (SQL/JSON Path)
JSON_VALUE(data, '$.user.name')

-- PostgreSQL (传统操作符，16 之前的主要方式)
data -> 'user' ->> 'name'          -- 操作符链
data #>> '{user,name}'             -- 路径操作符
jsonb_path_query(data, '$.user.name')  -- SQL/JSON Path (12+)

-- MySQL (操作符)
data -> '$.user.name'              -- 返回 JSON 值（带引号）
data ->> '$.user.name'             -- 返回文本值（无引号）
JSON_EXTRACT(data, '$.user.name')  -- 函数形式

-- SQL Server (完全不同的风格)
JSON_VALUE(data, '$.user.name')    -- 标量值
JSON_QUERY(data, '$.user')         -- 对象/数组
OPENJSON(data, '$.items')          -- 类似 JSON_TABLE 但语法不同

-- Oracle
JSON_VALUE(data, '$.user.name')    -- 最接近标准
data.user.name                     -- 点号路径语法 (21c+)
```

### JSON 存储模型差异

| 引擎 | 存储方式 | 优势 | 劣势 |
|------|---------|------|------|
| PostgreSQL (JSONB) | 二进制解析存储，去重键 | 索引支持强（GIN）、查询快 | 写入稍慢、不保留格式 |
| PostgreSQL (JSON) | 原始文本存储 | 保留格式 | 查询慢、不支持索引 |
| MySQL (JSON) | 二进制格式 | 部分路径更新 | 索引需要虚拟列 |
| Oracle | 多种（OSON 二进制 / 文本） | 21c+ 原生二进制性能好 | 旧版本性能一般 |
| SQL Server | 文本存储在 NVARCHAR | 简单 | 无专用索引、每次查询需解析 |
| ClickHouse | 原生 JSON 类型 (新) / String | 列式存储优化 | JSON 类型较新 |

## LISTAGG

LISTAGG 是一个将多行值连接为单个字符串的聚合函数：

```sql
-- SQL:2016 标准
SELECT department,
    LISTAGG(name, ', ') WITHIN GROUP (ORDER BY name) AS members
FROM employees
GROUP BY department;
-- 返回: 'Alice, Bob, Charlie'

-- 带去重 (SQL:2016)
SELECT department,
    LISTAGG(DISTINCT name, ', ') WITHIN GROUP (ORDER BY name) AS members
FROM employees
GROUP BY department;

-- 溢出处理 (SQL:2016)
SELECT department,
    LISTAGG(name, ', ' ON OVERFLOW TRUNCATE '...' WITH COUNT) AS members
FROM employees
GROUP BY department;
```

| 引擎 | LISTAGG | 替代函数 |
|------|---------|---------|
| Oracle | ✓ (11g R2+) | `WM_CONCAT`（已弃用） |
| SQL Server | ✗ | `STRING_AGG()` (2017+)、`FOR XML PATH` |
| PostgreSQL | ✗ | `STRING_AGG()` (9.0+)、`array_agg()` + `array_to_string()` |
| MySQL | ✗ | `GROUP_CONCAT()` |
| SQLite | ✗ | `GROUP_CONCAT()` |
| BigQuery | ✗ | `STRING_AGG()` |
| Snowflake | ✓ | `LISTAGG()` |
| ClickHouse | ✗ | `groupArray()` + `arrayStringConcat()` |

各引擎替代语法对比：

```sql
-- SQL:2016 标准
LISTAGG(name, ', ') WITHIN GROUP (ORDER BY name)

-- PostgreSQL / SQL Server 2017+ / BigQuery
STRING_AGG(name, ', ' ORDER BY name)      -- PG: ORDER BY 在括号内
STRING_AGG(name, ', ') WITHIN GROUP (ORDER BY name)  -- SS: 类似标准

-- MySQL / SQLite
GROUP_CONCAT(name ORDER BY name SEPARATOR ', ')

-- Oracle
LISTAGG(name, ', ') WITHIN GROUP (ORDER BY name)  -- 符合标准
```

## MATCH_RECOGNIZE（行模式识别）

MATCH_RECOGNIZE 允许在行序列上执行正则模式匹配，用于检测复杂的事件序列：

```sql
-- SQL:2016 标准: 检测股价"V 形反转"模式
SELECT *
FROM stock_prices
MATCH_RECOGNIZE (
    PARTITION BY symbol
    ORDER BY trade_date
    MEASURES
        FIRST(down.trade_date) AS start_date,
        LAST(up.trade_date)    AS end_date,
        FIRST(down.price)      AS start_price,
        LAST(up.price)         AS end_price
    PATTERN (down+ up+)
    DEFINE
        down AS price < PREV(price),
        up   AS price > PREV(price)
) AS mr;
```

| 引擎 | MATCH_RECOGNIZE |
|------|----------------|
| Oracle | ✓ (12c+) |
| Snowflake | ✓ |
| Trino/Presto | ✓ |
| MySQL | ✗ |
| PostgreSQL | ✗ |
| SQL Server | ✗ |
| BigQuery | ✗ |
| ClickHouse | ✗ |

> MATCH_RECOGNIZE 是一个非常强大但实现复杂的特性。只有 Oracle 和少数分析引擎支持。对于时序数据分析、欺诈检测、事件序列分析场景非常有价值。

## 多态表函数 (Polymorphic Table Functions, PTF)

```sql
-- SQL:2016 标准: 表函数的输出模式可以根据输入动态确定
-- 概念示例（标准语法非常学术化）
SELECT * FROM ptf_unpivot(
    TABLE orders,
    COLUMNS(q1, q2, q3, q4),
    'quarter', 'amount'
);
```

PTF 的实际采纳率极低。Oracle 21c 有实验性支持，其他引擎基本不支持。概念上很有趣，但标准定义过于抽象，各引擎更倾向于用私有语法实现类似功能（如 UNPIVOT、LATERAL 等）。

## 对引擎开发者的实现建议

1. **JSON 支持是当前最高优先级的标准特性**。按以下顺序实现：
   - 第一阶段：JSON 数据类型 + JSON_VALUE + JSON_EXTRACT（或 `->` 操作符）
   - 第二阶段：JSON_OBJECT + JSON_ARRAY + JSON_OBJECTAGG + JSON_ARRAYAGG
   - 第三阶段：JSON_TABLE（复杂但价值极高）
   - 第四阶段：JSON_EXISTS + IS JSON + 路径语言完整支持

2. **存储模型选择很关键**。
   - OLTP 引擎：推荐二进制存储（参考 PostgreSQL JSONB），支持部分更新和索引
   - OLAP 引擎：考虑将 JSON 展平为列式存储（参考 ClickHouse）
   - 混合方案：文本存储 + 计算列/虚拟列索引（参考 MySQL）

3. **同时支持标准函数和操作符语法**。用户习惯了 `->` / `->>` 操作符（更简洁），但标准 JSON_VALUE/JSON_QUERY 对工具生态兼容性更好。

4. **LISTAGG 或 STRING_AGG 二选一实现**。STRING_AGG 更简洁且被更多引擎采用（PG、SS、BigQuery），但如果兼容 Oracle 则需要 LISTAGG。两者可以都支持。

5. **MATCH_RECOGNIZE 面向特定场景**。如果你的引擎面向实时分析、事件处理、时序数据，MATCH_RECOGNIZE 是差异化特性。否则优先级低。

6. **PTF 可以跳过**。标准定义过于抽象，实际需求可以通过其他方式满足。
