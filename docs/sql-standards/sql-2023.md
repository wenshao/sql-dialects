# SQL:2023 标准

SQL:2023（ISO 9075:2023）是最新的 SQL 标准版本，发布于 2023 年 6 月。这个版本有两个重大方向：一是图查询（SQL/PGQ），将属性图查询能力直接嵌入 SQL；二是大量实用的小改进（ANY_VALUE、GREATEST/LEAST 等），这些在各引擎中早已有非标准实现，标准终于追上了现实。

## ANY_VALUE

ANY_VALUE 解决了 GROUP BY 中非聚合列的经典问题：

```sql
-- 问题: 查询每个部门的信息，但 manager_name 不在 GROUP BY 中
-- 传统 SQL 要求: SELECT 中非聚合列必须出现在 GROUP BY 中
-- MySQL sql_mode 不含 ONLY_FULL_GROUP_BY 时允许，但结果不确定

-- SQL:2023 标准
SELECT department_id,
    ANY_VALUE(department_name) AS dept_name,  -- 明确表示"取任意一个值"
    COUNT(*) AS emp_count
FROM employees
GROUP BY department_id;
```

| 引擎 | ANY_VALUE 支持 | 首次版本 |
|------|--------------|---------|
| MySQL | ✓ | 5.7 |
| PostgreSQL | ✗ | 不支持（可用 `MIN()` 或 `MAX()` 替代） |
| SQL Server | ✗ | 不支持 |
| Oracle | ✗ | 不支持（可用 `MIN()` 替代） |
| BigQuery | ✓ | 原生支持 |
| Snowflake | ✓ | 原生支持 |
| ClickHouse | ✓ | `any()` 函数 |
| SQLite | ✗ | 不支持（但 SQLite 本身允许非聚合列，取第一行的值） |

> MySQL 是 ANY_VALUE 的先驱，在 5.7 引入 `ONLY_FULL_GROUP_BY` 严格模式的同时提供了 ANY_VALUE 作为显式的"我知道结果不确定"声明。

## GREATEST / LEAST

返回参数列表中的最大值/最小值：

```sql
-- SQL:2023 标准
SELECT GREATEST(a, b, c) AS max_val,
       LEAST(a, b, c)    AS min_val
FROM t;

-- 常见用例: 限制值范围
SELECT LEAST(GREATEST(score, 0), 100) AS clamped_score FROM results;
```

| 引擎 | GREATEST/LEAST | NULL 处理 |
|------|---------------|----------|
| MySQL | ✓ | 任一参数为 NULL 则返回 NULL |
| PostgreSQL | ✓ | 忽略 NULL（除非全部为 NULL） |
| Oracle | ✓ | 任一参数为 NULL 则返回 NULL |
| SQL Server | ✓ (2022+) | 任一参数为 NULL 则返回 NULL |
| SQLite | ✓ | 忽略 NULL |
| BigQuery | ✓ | 任一参数为 NULL 则返回 NULL |
| Snowflake | ✓ | 任一参数为 NULL 则返回 NULL |
| ClickHouse | ✓ | `greatest()` / `least()` |

> **NULL 处理差异是一个陷阱**。PostgreSQL 和 SQLite 的行为（忽略 NULL）与标准和其他引擎（遇 NULL 返回 NULL）不同。SQL:2023 标准规定遇 NULL 返回 NULL。

```sql
-- PostgreSQL
SELECT GREATEST(1, NULL, 3);  -- 返回 3

-- MySQL / Oracle / SQL Server 2022+
SELECT GREATEST(1, NULL, 3);  -- 返回 NULL
```

## 数字字面量分隔符

```sql
-- SQL:2023 标准: 使用下划线作为数字分隔符，提高可读性
SELECT 1_000_000;           -- 一百万
SELECT 3.141_592_653;       -- 圆周率
SELECT 0xFF_FF;             -- 十六进制
```

| 引擎 | 数字分隔符 |
|------|----------|
| PostgreSQL | ✓ (16+) |
| MySQL | ✗ |
| Oracle | ✗ |
| SQL Server | ✗ |
| SQLite | ✗ |
| BigQuery | ✗ |
| Snowflake | ✗ |
| ClickHouse | ✗ |

> 目前只有 PostgreSQL 16+ 支持。这个特性在编程语言中很常见（Java、Python、Rust 等），但 SQL 引擎的采纳非常缓慢。

## SQL/PGQ: 图查询 (Property Graph Queries)

SQL:2023 中最具前瞻性的特性是 SQL/PGQ（ISO 9075-16），它将属性图查询直接嵌入到 SQL 中，无需使用独立的图数据库：

```sql
-- 第一步: 在关系表上定义属性图
CREATE PROPERTY GRAPH social_graph
    VERTEX TABLES (
        persons KEY (person_id)
            LABEL Person PROPERTIES (name, age)
    )
    EDGE TABLES (
        friendships KEY (friendship_id)
            SOURCE KEY (person_id_1) REFERENCES persons (person_id)
            DESTINATION KEY (person_id_2) REFERENCES persons (person_id)
            LABEL FriendOf PROPERTIES (since)
    );

-- 第二步: 使用图模式匹配查询
SELECT name, friend_name, since
FROM GRAPH_TABLE (social_graph
    MATCH (p:Person) -[f:FriendOf]-> (q:Person)
    WHERE p.name = 'Alice'
    COLUMNS (p.name, q.name AS friend_name, f.since)
);

-- 路径查询: 找到两个人之间的最短路径
SELECT *
FROM GRAPH_TABLE (social_graph
    MATCH SHORTEST (p:Person) -[f:FriendOf]->{1,6} (q:Person)
    WHERE p.name = 'Alice' AND q.name = 'Bob'
    COLUMNS (p.name AS start_name, q.name AS end_name)
);
```

### 图查询支持情况

| 引擎 | SQL/PGQ 支持 | 替代方案 |
|------|-------------|---------|
| Oracle | ✓ (23ai) | 最早实现的关系数据库 |
| PostgreSQL | 规划中 | Apache AGE 扩展（Cypher 语法） |
| SQL Server | ✗ | 图表功能（`MATCH` 语法，2017+，非标准） |
| MySQL | ✗ | 无 |
| DuckDB | 实验性 | 社区开发中 |
| 其他引擎 | ✗ | 使用独立图数据库（Neo4j、TigerGraph 等） |

> SQL/PGQ 的语法受 GQL（Graph Query Language, ISO 39075）影响，GQL 也是 2023 年发布的独立标准。两者共享查询语法的设计。

## JSON 增强

SQL:2023 在 SQL:2016 的 JSON 基础上进行了多项改进：

```sql
-- JSON 简化语法 (Simplified JSON Accessor)
-- 允许用点号路径直接访问 JSON，无需 JSON_VALUE
SELECT t.data.name,
       t.data.address.city
FROM documents t;

-- JSON_SERIALIZE: 将 SQL 值序列化为 JSON 字符串
SELECT JSON_SERIALIZE(json_col RETURNING VARCHAR(1000) FORMAT JSON);

-- JSON 构造器增强
SELECT JSON_OBJECT(
    'name': name,           -- 简化语法，不需要 VALUE 关键字
    'tags': JSON_ARRAY(tag1, tag2 NULL ON NULL)
) FROM products;

-- JSON 类型构造器（标准化 JSON 类型）
DECLARE @j JSON;            -- JSON 作为一等类型
```

JSON 简化访问语法目前只有 Oracle 21c+ 完整支持。

## 多维数组增强

```sql
-- SQL:2023 增强了 ARRAY 操作
SELECT ARRAY_AGG(name ORDER BY name) FROM employees;    -- 聚合为数组
SELECT TRIM_ARRAY(arr, 2) FROM t;                       -- 移除末尾元素
SELECT arr[1:3] FROM t;                                  -- 数组切片
```

| 函数 | PostgreSQL | BigQuery | Snowflake | ClickHouse |
|------|-----------|----------|-----------|------------|
| ARRAY_AGG | ✓ | ✓ | ✓ (ARRAY_AGG) | groupArray |
| 数组切片 | ✓ (`arr[1:3]`) | ✗ | ✓ (ARRAY_SLICE) | arraySlice |
| TRIM_ARRAY | ✗ | ✗ | ✗ | ✗ |
| ARRAY 构造器 | ✓ | ✓ | ✓ | ✓ |

## 其他实用新增

### UNIQUE NULLS DISTINCT / UNIQUE NULLS NOT DISTINCT

```sql
-- SQL:2023: 明确 UNIQUE 约束中 NULL 的处理方式
CREATE TABLE t (
    a INTEGER,
    UNIQUE NULLS NOT DISTINCT (a)  -- 多个 NULL 视为相同，只允许一个 NULL
);

-- 默认行为 (NULLS DISTINCT): 多个 NULL 视为不同，允许多个 NULL
CREATE TABLE t2 (
    a INTEGER,
    UNIQUE NULLS DISTINCT (a)
);
```

PostgreSQL 15+ 支持 `NULLS NOT DISTINCT` 选项。其他引擎的默认行为各不相同。

### 增强的类型转换

```sql
-- SQL:2023 中的布尔字面量增强
SELECT * FROM t WHERE flag IS TRUE;
SELECT * FROM t WHERE flag IS NOT FALSE;

-- 更宽松的隐式类型转换规则
```

## 对引擎开发者的实现建议

1. **优先实现 ANY_VALUE、GREATEST/LEAST**。这些函数实现成本极低，但用户需求很高。它们已经在多个引擎中以非标准形式存在多年，标准化只是追认现实。

2. **数字分隔符实现简单**。只需在词法分析器（lexer）中添加对下划线的处理即可，是一个很好的"快速胜利"。

3. **SQL/PGQ 是长期方向，短期不急**。图查询标准刚刚发布，生态尚不成熟。但如果你的引擎需要处理社交网络、知识图谱、供应链等图结构数据，可以开始调研。

4. **JSON 简化访问语法值得关注**。用 `t.data.name` 代替 `JSON_VALUE(t.data, '$.name')` 大幅提升可读性。如果你的引擎已有 JSON 支持，这是一个高价值的增强。

5. **NULLS NOT DISTINCT 是一个实用的小特性**。如果你的引擎支持 UNIQUE 约束，可以考虑加入这个选项。它解决了一个长期的歧义问题。

6. **关注标准演进方向**。SQL:2023 显示标准委员会越来越重视"追认广泛实践"（ANY_VALUE、GREATEST/LEAST）而非"超前定义"（UDT、PTF）。这对引擎开发者意味着：实现已有广泛需求的特性，回报率最高。
