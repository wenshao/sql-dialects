# SQL-86 / SQL-89 标准

SQL-86（ISO 9075:1987）是第一个正式的 SQL 标准，由 ANSI 于 1986 年发布、ISO 于 1987 年采纳。SQL-89 是一个小幅修订版本，主要增加了引用完整性约束。由于两者内容相近，本文合并介绍。

## 历史背景

- 1970 年 E.F. Codd 发表关系模型论文
- 1974 年 IBM System R 项目开始使用 SEQUEL（后改名 SQL）
- 1979 年 Oracle V2 成为第一个商业 SQL 产品
- 1986 年 ANSI 发布 SQL-86（X3.135-1986）
- 1989 年发布 SQL-89 修订版（增加完整性约束）

## SQL-86 核心特性

### 数据操作语言（DML）

SQL-86 标准化了四个基本操作：

```sql
-- SELECT: 基本查询
SELECT col1, col2 FROM table1 WHERE col1 = 'value';

-- INSERT: 插入数据
INSERT INTO table1 (col1, col2) VALUES ('a', 'b');

-- UPDATE: 更新数据
UPDATE table1 SET col1 = 'new_value' WHERE col2 = 'condition';

-- DELETE: 删除数据
DELETE FROM table1 WHERE col1 = 'value';
```

**支持矩阵：**

| 特性 | MySQL | PostgreSQL | Oracle | SQL Server | SQLite | BigQuery | Snowflake | ClickHouse |
|------|-------|-----------|--------|-----------|--------|----------|-----------|------------|
| SELECT | 完整 | 完整 | 完整 | 完整 | 完整 | 完整 | 完整 | 完整 |
| INSERT | 完整 | 完整 | 完整 | 完整 | 完整 | 完整 | 完整 | 完整 |
| UPDATE | 完整 | 完整 | 完整 | 完整 | 完整 | 有差异 | 完整 | 有限 |
| DELETE | 完整 | 完整 | 完整 | 完整 | 完整 | 有差异 | 完整 | 有限 |

> BigQuery 的 UPDATE/DELETE 需要使用 `WHERE` 子句且只能操作分区列或 `WHERE TRUE`。ClickHouse 的 UPDATE/DELETE 是异步 mutation 操作，不是即时的。

### WHERE 子句与比较运算

```sql
-- 标准比较运算符
SELECT * FROM t WHERE a = 1;
SELECT * FROM t WHERE a <> 1;   -- 不等于
SELECT * FROM t WHERE a > 1;
SELECT * FROM t WHERE a >= 1;
SELECT * FROM t WHERE a < 1;
SELECT * FROM t WHERE a <= 1;

-- 逻辑运算符
SELECT * FROM t WHERE a = 1 AND b = 2;
SELECT * FROM t WHERE a = 1 OR b = 2;
SELECT * FROM t WHERE NOT (a = 1);

-- BETWEEN, IN, LIKE, IS NULL
SELECT * FROM t WHERE a BETWEEN 1 AND 10;
SELECT * FROM t WHERE a IN (1, 2, 3);
SELECT * FROM t WHERE a LIKE 'abc%';
SELECT * FROM t WHERE a IS NULL;
```

所有主流引擎均完整支持以上语法。`!=` 是 `<>` 的非标准别名，但几乎所有引擎都接受。

### 聚合函数

```sql
SELECT COUNT(*), SUM(amount), AVG(amount), MAX(amount), MIN(amount)
FROM orders
GROUP BY customer_id
HAVING COUNT(*) > 5;
```

SQL-86 标准化了 `COUNT`、`SUM`、`AVG`、`MAX`、`MIN` 五个聚合函数，以及 `GROUP BY` 和 `HAVING` 子句。

### ORDER BY

```sql
SELECT col1, col2 FROM t ORDER BY col1 ASC, col2 DESC;
```

所有引擎均支持。注意 SQL 标准要求 `ORDER BY` 中的列必须出现在 `SELECT` 列表中（SQL-86 严格要求），但现代引擎普遍放宽了这个限制。

### 数据定义语言（DDL）

```sql
-- 创建表
CREATE TABLE employees (
    emp_id    INTEGER       NOT NULL,
    name      CHAR(30)      NOT NULL,
    dept_id   INTEGER
);

-- 删除表
DROP TABLE employees;
```

SQL-86 中 `CREATE TABLE` 功能非常基础，仅支持 `NOT NULL` 约束。数据类型只有：`INTEGER`、`SMALLINT`、`NUMERIC`、`DECIMAL`、`FLOAT`、`REAL`、`DOUBLE PRECISION`、`CHAR`。

### 数据类型对比

| SQL-86 类型 | MySQL | PostgreSQL | Oracle | SQL Server | SQLite |
|------------|-------|-----------|--------|-----------|--------|
| INTEGER | INT / INTEGER | INTEGER | NUMBER(38) | INT | INTEGER |
| SMALLINT | SMALLINT | SMALLINT | NUMBER(38) | SMALLINT | INTEGER |
| NUMERIC(p,s) | NUMERIC(p,s) | NUMERIC(p,s) | NUMBER(p,s) | NUMERIC(p,s) | NUMERIC |
| DECIMAL(p,s) | DECIMAL(p,s) | DECIMAL(p,s) | NUMBER(p,s) | DECIMAL(p,s) | REAL |
| FLOAT | FLOAT | FLOAT8 | FLOAT | FLOAT | REAL |
| REAL | FLOAT | REAL | FLOAT | REAL | REAL |
| DOUBLE PRECISION | DOUBLE | DOUBLE PRECISION | FLOAT | FLOAT(53) | REAL |
| CHAR(n) | CHAR(n) | CHAR(n) | CHAR(n) | CHAR(n) | TEXT |

> Oracle 没有原生 INTEGER 类型，`INTEGER` 被映射为 `NUMBER(38)`。SQLite 使用动态类型系统，类型名仅作为亲和性提示。

## SQL-89 新增特性

### 引用完整性约束

SQL-89 最重要的增量是引入了完整性约束：

```sql
CREATE TABLE departments (
    dept_id   INTEGER   NOT NULL  PRIMARY KEY,
    name      CHAR(50)  NOT NULL  UNIQUE
);

CREATE TABLE employees (
    emp_id    INTEGER   NOT NULL  PRIMARY KEY,
    name      CHAR(50)  NOT NULL,
    dept_id   INTEGER   REFERENCES departments(dept_id),
    salary    DECIMAL(10,2) DEFAULT 0,
    CHECK (salary >= 0)
);
```

**约束支持矩阵：**

| 约束 | MySQL | PostgreSQL | Oracle | SQL Server | SQLite | BigQuery | Snowflake | ClickHouse |
|------|-------|-----------|--------|-----------|--------|----------|-----------|------------|
| NOT NULL | 完整 | 完整 | 完整 | 完整 | 完整 | 完整 | 完整 | 完整 |
| PRIMARY KEY | 完整 | 完整 | 完整 | 完整 | 完整 | 不强制 | 不强制 | 排序键 |
| UNIQUE | 完整 | 完整 | 完整 | 完整 | 完整 | 不强制 | 不强制 | 不支持 |
| FOREIGN KEY | InnoDB | 完整 | 完整 | 完整 | 可选 | 不强制 | 不强制 | 不支持 |
| CHECK | 8.0.16+ | 完整 | 完整 | 完整 | 完整 | 不支持 | 不支持 | 不支持 |
| DEFAULT | 完整 | 完整 | 完整 | 完整 | 完整 | 完整 | 完整 | 完整 |

关键差异说明：

- **MySQL**：CHECK 约束在 8.0.16 之前被解析但被静默忽略；FOREIGN KEY 仅 InnoDB 引擎支持
- **SQLite**：外键默认关闭，需要 `PRAGMA foreign_keys = ON` 显式启用
- **BigQuery / Snowflake**：主键和外键是声明式的（信息性），不强制执行，用于查询优化提示
- **ClickHouse**：作为列式分析引擎，不支持传统约束模型

### 标准与现实的差距

SQL-86/89 虽然是最基础的标准，但即便到今天，各引擎在某些基础行为上仍有差异：

**1. NULL 的三值逻辑**

标准定义 NULL 参与比较时返回 UNKNOWN，但各引擎在边界情况下行为不一致：

```sql
-- 标准: NULL = NULL 返回 UNKNOWN (不是 TRUE)
SELECT * FROM t WHERE NULL = NULL;  -- 应返回空结果

-- 但在 UNION、DISTINCT、GROUP BY 中，NULL 被视为相等
SELECT DISTINCT col FROM t;  -- 多个 NULL 只保留一个
```

**2. 字符串比较的尾部空格处理**

```sql
-- CHAR(10) 类型存储 'abc' 时会补空格为 'abc       '
-- 'abc' = 'abc   ' 是否为 TRUE？
-- SQL 标准: CHAR 比较忽略尾部空格 → TRUE
-- MySQL: 取决于 collation（默认忽略）
-- PostgreSQL: CHAR 忽略，VARCHAR 不忽略
-- Oracle: CHAR 忽略，VARCHAR2 不忽略
```

**3. 标识符大小写**

```sql
-- SQL 标准: 无引号标识符折叠为大写
CREATE TABLE Employees ...;  -- 标准视为 EMPLOYEES
-- Oracle: 折叠为大写 ✓
-- PostgreSQL: 折叠为小写 ✗
-- MySQL: 取决于 lower_case_table_names 参数和操作系统
-- SQL Server: 取决于 collation
```

## 对引擎开发者的实现建议

1. **SQL-86/89 是必须完整实现的底线**。所有 DML 和基本 DDL 必须正确实现，这是用户的最低期望。
2. **约束实现可以分阶段**。优先实现 NOT NULL 和 PRIMARY KEY，然后是 UNIQUE 和 DEFAULT，最后是 FOREIGN KEY 和 CHECK。分析型引擎可以只做声明式约束（解析但不强制执行）。
3. **NULL 处理必须从一开始就正确**。三值逻辑渗透到 SQL 的每个角落（WHERE、JOIN、聚合、DISTINCT、ORDER BY），后期修复代价极大。
4. **字符集/排序规则要早期决策**。它影响字符串比较、LIKE、ORDER BY、UNIQUE 索引等行为。
5. **标识符处理**。建议遵循 PostgreSQL 的做法（折叠为小写），因为现代应用普遍使用小写标识符。但如果目标是兼容 MySQL 或 Oracle，则需要不同的策略。
